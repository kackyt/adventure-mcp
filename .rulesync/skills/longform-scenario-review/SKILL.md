# Longform Scenario Review

長編アドベンチャー（章をまたぐ作品・推理もの）の Ink シナリオを、**作る側とは別の目**で点検する「壊しにかかる」スキル。作成は `ink-scenario-creator`、レビューはこちらが担当する。

## 前提・なぜ別スキルか

同一の生成パスでの自己採点は確証バイアスで弱い。**書いた文脈と切り離した独立レビュー**ほど効く。機械テストで取れる客観項目と、ルーブリック照合で取れる論理・視点・手続きの項目を分け、**人間の最終関所の前段**として違反候補を洗い出す。

## 手順

1. **ルーブリックを読み込む**：[ink-scenario-creator/references/longform_mystery_craft.md](../ink-scenario-creator/references/longform_mystery_craft.md) の「レビュー・ルーブリック」と本文 A〜C を観点として使う（[puzzle_design.md](../ink-scenario-creator/references/puzzle_design.md) §5-6 も併用）。
2. **対象を素読み**：対象 `.ink`（と必要ならテスト）を通読する。**封印層（真相）に引きずられず**、プレイヤーが各章で実際に得る情報の順で読む。
3. **機械検証を確認**：`pnpm run build:ink` が通るか、シナリオテストが緑か（一意性・ソフトロック無・章ゲート・到達性）。落ちていれば最優先で報告。
4. **ルーブリックを1項ずつ採点**（下記）。各指摘は **`file:line` ＋ 該当箇所の引用 ＋ なぜ違反か ＋ 直し方の方向**をセットで。憶測でなく本文の根拠を示す。
5. **重大度を付ける**：Blocker（因果破綻・ソフトロック・前出し/後出し・スパイン齟齬）／Major（視点崩壊・早期解決・メタ採点）／Minor（用語・語り口の揺れ）。
6. **独立性を上げる場合**：観点が多い・大規模なときは、観点別（因果／視点／手続き／配分／ネタばらし／機械検証）に**サブエージェントへ分担**してよい（ユーザーが許せば）。レビューはあくまで指摘までで、勝手に本文を書き換えない（修正は作成側の判断で）。

## ルーブリック（各項 yes/no ＋ 違反は file:line）

- **視点**：GM/クイズ司会の口調・出題・採点が無い／地の文は主人公の内面・観察か。
- **選択肢**：取得後に消える選択肢が無いか（進行必須の誘導を除く）。
- **NG ルート**：メタ採点（「空振り」「決め手にならない」）でなく NPC の反応で示しているか。
- **因果の鎖**：各章に固有の目的があり、その章で掴んだ情報が次章の動機になっているか／飛躍・無関係挿話・封印層との齟齬が無いか。
- **フラグ**：結末を意味的に分岐させるゲートか（飾りでないか）／必須と任意（真EDの鍵）が分離しているか。
- **ネタばらし**：結論・固有名・真相・深いフラグの早出しが無いか。
- **手続き×立場**：手続きが生む前に断定していないか（前出し）／有能な手続きが見落とすはずのない物を後章で発見していないか（後出し）／主人公の役割・権限・アクセスを超えて知っていないか／解法が有能な手続きで崩れないか。
- **配分**：中核を早く解きすぎていないか／手がかりの強さが狙ったペースに合っているか。
- **用語・制度**：役割・権限・段取りの語が正確か。
- **帰納/推理**：一本道○×でないか／誤答が成立する余地があるか（フーダニットを決め打ちにしていないか）。
- **制約遵守**：キャラ・道具・装置が確立した知識/能力/時系列を超えて万能化していないか／道具で推理を飛ばしていないか／前提を場面都合で勝手に緩めていないか。
- **検証**：ビルド＋テストが緑か。

## 出力

重大度順に指摘を列挙し、最後に「人間が最終確認すべき点（味・リアリティ・因果の納得感）」を 1-3 点添える。**修正の実施は依頼されたときのみ**行い、既定では指摘に留める。
