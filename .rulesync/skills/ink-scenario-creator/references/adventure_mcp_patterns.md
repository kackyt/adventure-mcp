# adventure-mcp プロジェクト固有の実装パターン

`adventure-mcp` プロジェクトにおけるシナリオ作成のベストプラクティスと制約事項です。

## 基本原則：状態管理の秘匿

AIエディタ（MCP）とシナリオエンジン（InkJS）は分離されています。AIは「ゲーム全体のシナリオ」や「未解明のフラグ」を事前に知ることはなく、現在プレイヤーがいる場所、見えているもの、実行可能なアクションのみを提供されます。

- **AIに考えさせない**: プレイヤーのステータス増減や、アイテム取得の判定、イベント進行のフラグ管理はすべて `.ink` ファイル内に記述します。AIに「アイテムを与えて」と指示するのではなく、Ink内で完結させてください。
- **堅牢な状態遷移**: シナリオ上の分岐は必ずInkの選択肢（Choices）として表現します。AIはテキストの提示と、ユーザーの意思決定に基づく選択肢の実行のみを仲介します。

## 公開ステータス変数 `public_status`

AI（ゲームマスター役）には未解明のフラグやネタバレを渡しませんが、HP・所持金・現在地のような「プレイヤーに常時見えていてよいステータス」だけは予約変数 `public_status` で**明示的に公開**します。生成する `.ink` には**必ず** `public_status` を含めてください。

### 書式

ファイル先頭で、固定名 `public_status` を**文字列**として宣言し、公開する変数名を**カンマ区切り**で列挙します。

```ink
VAR player_hp = 20
VAR gold = 0
VAR location = "entrance"
VAR has_master_key = false       // 解法フラグ：公開しない

// player_hp / gold / location のみ公開。has_master_key は含めない。
VAR public_status = "player_hp,gold,location"
```

- 変数名は `public_status` 固定（リネーム不可）。値は文字列。
- 列挙する名前は `VAR` 宣言済みの変数名と完全一致させ、余分な空白を入れない。
- 公開するものが無いシナリオでも `VAR public_status = ""` を必ず宣言する。
- `public_status` 変数自体と、列挙していない変数は AI／プレイヤーへの出力に含まれない（エンジンが除外）。AI に渡るのは列挙した各変数の現在値だけ。

### 見せる変数／隠す変数の判断基準

| 区分 | 例 | `public_status` に載せる？ |
| --- | --- | --- |
| 進行状況の指標 | `player_hp`, `gold`, `location`, `turn_count` | ○ 載せる |
| 解法・進行フラグ | `has_key`, `is_door_open`, `knows_culprit`（`has_*` / `is_*` / `knows_*` 系） | × 載せない |
| 真相・正解値 | `culprit_name`, `correct_code` | × 載せない |

判断に迷ったら「**この値をプレイヤー画面に常時表示しても謎の難易度が変わらないか？**」を基準にします。難易度が変わる＝ネタバレなので載せません。

## 出力ファイルの構成と配置場所

- 作成したシナリオファイルは、常に `engine/assets/` ディレクトリ配下に `.ink` 形式で保存します。
- 保存後、プロジェクトルートまたは `engine/` ディレクトリにて `pnpm run build:ink` コマンドを実行し、コンパイルエラー（文法エラーや無効な参照など）がないことを確認してください。

## シナリオ記述のベストプラクティス

### 1. 状況の明確な描写
AIがゲームマスターとして振る舞うため、ノットの冒頭では「現在の状況、目に見えるもの」を明確に描写してください。

```ink
=== room_01 ===
あなたは薄暗い石造りの部屋にいる。
北には重厚な鉄の扉があり、部屋の中央には古びた宝箱が置かれている。
```

### 2. 選択肢は行動として提示
プレイヤーの取り得る行動を明確な選択肢として列挙します。一度しか実行できないアクション（宝箱を開ける等）には `*` を、何度でも実行できるアクション（調べる等）には `+` を使用します。

```ink
+ [周囲を調べる]
    壁には古い松明がかけられているが、火は点いていない。
    -> room_01
* [宝箱を開ける]
    宝箱には鍵がかかっておらず、中から古びた鍵を見つけた。
    ~ has_key = true
    -> room_01
```

### 3. デッドエンド（行き止まり）の回避
シナリオ内でプレイヤーが選択肢を失い、進行不能になる状態（デッドエンド）を避けるため、ノットの最後には必ず選択肢を配置するか、別のノットへ遷移（`->`）するか、あるいはシナリオ終了（`-> DONE`）を明記してください。

### 4. フォールバック選択肢（Fallback Choices）
状況が変化した際に、何も選択できるアクションがなくなってしまうことを防ぐため、条件なしのフォールバック選択肢を用意することが有効な場合があります。

```ink
* [調べる] -> investigate
* [休む] -> rest
+ ->
    （これ以上できることはないようだ…）
    -> DONE
```

## InkJS / CLI 検証で踏んだ落とし穴（実証済み）

`engine`（inkjs 2.x）でのコンパイル・実機プレイで実際に遭遇した不具合とその回避策です。新規シナリオでも繰り返し踏みやすいので必ず確認してください。

### 1. 同一ノットへの自己 divert では訪問回数が増えない
「部屋にとどまってアクションを繰り返す」ループを `-> same_knot` で実装すると、**そのノットの訪問回数（`{knot}` / `CNT?`）が再入場で増えません**（inkjs の既知の癖。`docs/06-reference/inkfiles/builtins/visit_count_bug_due_to_nested_containers.ink` 相当）。

そのため「初回だけ戦闘」「初回だけの描写」を `{ knot == 1: ... }` で制御すると、再訪のたびに再発します。

- **対策**: 自己ループする部屋の一度きりイベントは、訪問回数ではなく **明示的な真偽フラグ**で管理する。

```ink
VAR goblin_beaten = false
=== floor_2 ===
{ not goblin_beaten:
    -> combat ->
    ~ goblin_beaten = true
}
// 選択肢で -> floor_2 にループしても再発しない
```

訪問回数・gather 読み取り回数が確実に効くのは、**外部から毎回入り直すノット**（tunnel 呼び出し・thread・別ノットからの divert）や **gather ラベル**に対してです。

### 2. 選択肢リストの中に入れ子選択肢を直書きしない
ノット直下の選択肢（`*` / `+`）の配下に、さらにインデントして入れ子の選択肢を書くと、weave のギャザー構造が壊れ、再入場時に実行時エラー **「ran out of content. Do you need a '-> DONE' or '-> END'?」** が出ることがあります。

- **対策**: 入れ子の分岐は**独立したノットに切り出して `->` で飛ばす**（プロジェクト指針の「分岐はノットで」とも合致）。

```ink
// NG: floor_1 の選択肢リスト内に入れ子を直書き
+ [井戸を覗く]
    * [縄で降りる] -> floor_1
    * [素手で降りる] -> game_over

// OK: 独立ノットに切り出す
+ [井戸を覗く] -> well
=== well ===
* {carries(rope)} [縄で降りる] -> floor_1
* [素手で降りる] -> game_over
+ [やめておく] -> floor_1
```

### 3. インライン条件は「2分岐まで」かつ代入（`~`）を含められない
`{ cond: A | B }` のインライン条件は分岐が**最大2つ**まで。3分岐以上の出し分けや switch は**複数行ブロック**で書きます。また `{ cond: ~ var = x }` のようにインライン条件内へ `~` の代入文は置けません（コンパイルエラー）。

```ink
// 3分岐以上 / 代入を伴う条件は複数行で
{
- gate == sealed: 扉は封印されている。
- gate == runed:  ルーンが光っている。
- else:           扉は開いている。
}
{ hp <= 0:
    ~ death_cause = "sage"
    -> game_over
}
```

### 4. 移動・再入場の選択肢は sticky（`+`）にする
フロア間移動や「もう一度話す」など**繰り返し選べるべき選択肢を once-only（`*`）にすると、一度使った時点で消えてデッドエンド化**します（例: 一階に戻ると「上の階へ」が消える）。アイテム取得や一方向の状態遷移だけ `*`、移動・再入場系は `+` を使い分けてください。

### 5. tunnel 復帰後は「成功」と「中断（逃走）」をフラグで区別する
tunnel（`-> combat ->`）は勝利でも逃走（`->->`）でも同じ場所に戻るため、戻り先で結果を判別できません。撃破フラグを戦闘ノット末尾で一律に立てると「逃げても撃破扱い」になり、報酬や再戦の整合が崩れます。

- **対策**: 勝利時のみ立てるフラグ（例 `last_battle_won`）を戦闘の `win` gather 内でセットし、呼び出し側でそれを見て分岐する。

### 検証手順の推奨
`pnpm run build:ink` のコンパイル成功だけでは実行時エラー（上記 1・2 など）は検出できません。**生成 JSON を inkjs で実際に再生し**、一度きりイベントの再発・デッドエンド・ゲームオーバー分岐・クリア到達可否までスモークプレイで確認してください。なお生成 JSON 先頭に BOM が付くため、`Story` へ渡す前に除去します（`readFileSync(...,"utf-8").replace(/^﻿/, "")`）。
