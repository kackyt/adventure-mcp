# 謎解き要素の設計（adventure-mcp）

「お使い」ではなく**プレイヤーが推理して解く謎**を、状態秘匿アーキテクチャを壊さずに作るための定義・パターン・難易度レバー。

> 難易度モデルの2軸・合成則（加算 vs 乗算）・ジャンル別カタログ（嘘つき論理／アリバイ崩し／水平思考／換字暗号／論理グリッド 等）と対応 PoC の一覧は、総合リファレンス `docs/06-reference/PUZZLE_DESIGN.md` を参照。本書はその要約と Ink 実装の手引き。

> **最優先原則**: 「答えを N 択で選ぶゲーム」にしないこと。アドベンチャーの核は、世界をいじり試行錯誤しながら手がかりを見つける**過程**にある。解は「宣言」でなく**世界への働きかけで実演**させる（証拠を相手に突きつける／仕掛けを操作する）。どのジャンルでもこの探索の手応えを失わない。詳細は `docs/06-reference/PUZZLE_DESIGN.md` §0。

## 1. 「謎」の操作的定義（謎 ≠ お使い）

ある仕掛けが「謎」であるためには、次の3条件を**すべて**満たす。

1. **複数要素の統合 or 非自明な順序**
   単一行動では解けず、2つ以上の別々に発見する手がかり/アイテムの組み合わせ、または非自明な手順を要する。
2. **ノンスポイラー**
   正解の選択肢ラベル自体が答えを明かさない（後述の (B)(C) で実現）。
3. **失敗可能かつフェア**
   素朴に進むと解けず足踏み・別ルートになり得るが、提示済みの手がかりから**論理的に解ける**（理不尽な総当たり強制をしない）。

### 反例と正例（同梱サンプル）

| ファイル | 区分 | 理由 |
| --- | --- | --- |
| `engine/assets/room_escape.ink` | ❌ お使い | 引き出しを調べれば鍵が手に入り扉が開く。推理も失敗もない（条件1・3を満たさない）。 |
| `engine/assets/sages_tower.ink` のリドル | ❌ 総当たり可 | 答え（松明/太陽/灯火）を選択肢に並べており、条件2（ノンスポイラー）を満たさない。 |
| `engine/assets/frozen_key.ink` | ✅ 謎（簡単） | 別々に得た松明＋油を「油→点火」の順で使う。答えはラベルに書かれない。素手/松明のみは失敗。 |
| `engine/assets/frozen_key_normal.ink` | ✅ 謎（普通） | 上記に探索・条件充足・ミスリード・バッドエンド分岐を足してレバーを一段上げた版。 |

## 2. ノンスポイラーのコアモデル

選択肢メニュー＋状態秘匿＋サーバ側バリデーションという制約下では、**答えを選択肢に並べた瞬間にネタバレ（総当たり可能）**になる。これを避ける2モデルを基本とする。

### (B) 創発パズル（組み合わせ・順序）

個々の選択肢はすべて正当な行動。謎の本体は「どの行動を・何を持って・どの順で」行うか。ラベルは答えを言わない。

```ink
// 「油をかける」「松明をかざす」は行動を述べるだけ。「氷を溶かす正解」とは書かない。
// 正解は "油 → 点火" の順序（松明だけでは火力不足）= プレイヤーが推理する。
+ {has_oil && not oil_applied && not ice_melted} [氷に油をかける]
    とろりとした油が氷塊の表面いっぱいに広がった。
    ~ oil_applied = true
    -> altar_room
+ {has_torch && not ice_melted} [松明をかざす]
    {
    - oil_applied:
        油に一気に引火し、氷は見る間に溶け落ちた。中から鍵が現れた。
        ~ ice_melted = true
    - else:
        炎は弱々しく、表面がわずかに濡れただけだ。もっと強い火が要る。   // フェアな失敗ヒント
    }
    -> altar_room
```

### (C) 手がかりゲート（条件付き出現）

正解の選択肢は、必要な手がかり/アイテムが揃うまで `{条件}` で**出現させない**。メニューに答えが漏れない。

```ink
// 「鍵を取る」は氷が溶けるまでメニューに現れない。
+ {ice_melted && not has_key} [現れた鍵を取る]
    溶けた水の中から、濡れた鍵を拾い上げた。
    ~ has_key = true
    -> altar_room
```

> **(A) 完全一致の自由入力（暗証番号・固定キーワード等）はスコープ外。** 選択肢ではない入力経路の新設はエンジン/MCP 拡張を要するため、別途 Issue #13 で扱う。曖昧な自然言語リドル（同義語・表記ゆれ判定）は採用しない。

## 3. 謎のアーキタイプ・カタログ

| アーキタイプ | 概要 | Ink パターンの要点 |
| --- | --- | --- |
| **アイテム使用（手がかり統合型）** | 複数アイテムを揃え/組み合わせて使う | アイテム所持フラグ or `LIST` で管理し、`{carries(x) && carries(y)}` で正解行動をゲート |
| **手順・順序（状態機械）** | 正しい順序を踏むと状態が進む | `LIST State = a, b, c` で遷移を表し、各遷移を `{state == a}` でゲート。順序はヒントから推理させる |
| **条件充足** | 複数の前提が揃って初めて道が開く | `{cond_a && cond_b} [扉を開ける]` のように複合条件でゲート |
| **探索・観察** | 手がかり/アイテム自体が隠れている | 「調べる」行動で初めて入手（`frozen_key_normal` の床石下の油）。単体では謎にならず他型と組み合わせる |

> 単純な key-lock（`room_escape` の「調べる→鍵→開ける」）は**謎に含めない**。手がかりの統合も非自明な順序もなく、定義の条件1・3を満たさないため。

### アイテム管理の指針

`LIST` ＋ `carries()` 関数方式（`sages_tower.ink` 参照）と、用途が少なければ真偽フラグ方式（`frozen_key.ink` 参照）のどちらでもよい。**自己ループする部屋の一度きり判定は訪問回数に頼らず明示フラグで行う**（`adventure_mcp_patterns.md` の落とし穴1を参照）。

## 4. 謎/分岐軸の難易度レバー（簡単 / 普通 / 難しい）

謎の"構造"は (B)(C) で共通のまま、次の4レバーで難易度を段階化する。

| レバー | 簡単 | 普通 | 難しい |
| --- | --- | --- | --- |
| **ヒント密度** | ほぼ答え（手厚い） | 事実のみ・要約1回 | 暗示のみ／自力推理 |
| **ミスリード(red herring)** | 0 | 1〜2 | 3以上 |
| **謎の数** | 1〜3 | 4〜7 | 8以上 |
| **分岐の深さ** | ほぼ一本道 | 2〜3層・バッドエンド1〜2 | 4層以上・複数エンディング |

> ボリューム・戦闘・ゲームオーバー/バッドエンドの各軸レバーは別途（難易度カタログ Issue #11）で扱う。ここでは謎/分岐軸のみ。

### レバーの上げ下げ：Ink 差分例

**ヒント密度（手厚い → 事実のみ）** — 結論を述べず、事実だけ提示して推理させる。

```ink
// 簡単: 答えに近い
+ [氷塊を調べる]
    かなり分厚い氷だ。生半可な熱では、とても溶けそうにない。     // "熱が要る" と明言

// 普通: 事実のみ
+ [氷塊を調べる]
    白く凍てついた厚い氷だ。叩くと硬く高い音が返ってくる。       // 結論は言わない
```

**ミスリード（0 → 1〜2）** — それらしい誤答を置く。ただしフェアに「効かない」と示し、進行不能にしない。

```ink
+ {has_hammer && not ice_melted} [金槌で氷を打ち砕く]
    渾身の一撃を叩き込む。だが氷は鈍く撥ね返すばかりで、罅ひとつ入らない。   // 物理は無効と明示
    -> ice_chamber   // 失敗だがデッドエンドにしない
```

**分岐の深さ（バッドエンドの追加）** — 選択で結末を分岐させる。バッドエンドも必ず終端ED（`-> DONE`/`-> END`）に繋ぎ、ソフトロックにしない。致死/取り返しのつかない分岐は**予兆を必ず描写**する。

```ink
=== ignite ===
広げた液体は思いのほか多い。勢いよくやれば早いが、跳ね返る炎が危うい。   // 予兆（フェア）
* [慎重に炙る]
    ~ ice_melted = true
    -> ice_chamber
* [一息に引火させる]
    跳ね返った炎に腕を焼かれてしまった。
    ~ ice_melted = true
    ~ burned = true   // 結末分岐用フラグ。後の ending で文面を変える
    -> ice_chamber
```

## 5. 検証（必須）

`pnpm run build:ink` のコンパイル成功だけでは実行時の破綻（デッドエンド・一度きりイベント再発・解けない謎）を検出できない。**生成 JSON を inkjs で実際に再生**し、最低限以下を確認する（BOM 除去 `replace(/^﻿/, "")` を忘れない）。

- 手がかりから正解ルートに到達でき、クリア（`-> DONE`/`-> END`）できる
- 素朴に進む「失敗ルート」が存在し、かつそこから挽回できる（デッドエンドにならない）
- バッドエンドがある場合、意図どおり発生し、それも終端ED（ソフトロックでない）
- 一度きりイベントが再訪で再発しない

同梱の `engine/src/domain/services/frozen-key*.smoke.spec.ts` が、上記をスモークプレイで検証する実例。新しい謎シナリオを作ったら、同様のスモークプレイテストを添えること。
