# Ink Scenario Creator

このスキルは、`adventure-mcp` プロジェクトにおける Ink シナリオ作成を支援します。

## ワークフロー

### 0. 作成モードの選択（最初に必ず1問だけ）

シナリオ作成を始める前に、**作成モードを1問だけ確認**します（これ以外の質問の有無は選んだモードに従う）。あわせて「ご自身でプレイしますか？」も同じ問いに束ねて聞きます（ネタバレ開示の可否＝後述）。

- **おまかせモード** — ユーザーは方針（テーマ・難易度・ボリューム）だけ渡し、**ゲームも物語も AI が書く**。未指定の軸はプリセット既定値「普通」で補完して草案まで一気に作る。モード選択時に「拾えたテーマの要約＋既定値（難易度=普通・ボリューム=中）」を提示し、ユーザーがインフォームドに選べるようにする。
- **共作モード** — **物語はユーザーと作り込み**、AI は**ゲーム機構**（謎の成立・難易度バランス・分岐配線・スモークプレイ）を担当する。物語レイヤー＝ユーザー主導／ゲームレイヤー＝AI主導。

ネタバレ開示は**モードでなくレイヤーの属性**として扱う：物語（誰が書くか）と、ゲーム解法の開示（プレイするなら伏せる／しないなら全開示）は独立。「伏せる」を選んだら、謎は抽象レベルでのみ相談し、フェアさ・可解性はスモークプレイ＋（必要なら別エージェントの）コールドプレイで AI が代理保証する。

> 「ゲームの面白さ ≠ シナリオ（物語）の面白さ」。前者は AI＋検証で担保できるが、後者と**難易度・物語の整合性の最終判断は人間が原理的に必要**（AI は自分の難易度・整合性を自己採点しにくい＝知識の呪い）。これが共作モードと §6 の整合チェック関所の根拠。

### 1. 要件の理解と設計

提示されたアイデアやプロットをもとに、必要なフラグ（変数）やノット（場面）を設計します。adventure-mcp では AI はゲーム状態を直接保持せず、**すべての状態遷移・フラグ管理は Ink 内で完結**させます。制約とベストプラクティスは [adventure_mcp_patterns.md](./references/adventure_mcp_patterns.md) を参照。

「お使い」ではなく**プレイヤーが推理して解く謎**を入れる場合は [puzzle_design.md](./references/puzzle_design.md) を参照。特に：

- **難易度は「再認/選択で済むもの＝易しい」「生成（自分で着想・法則・統合を作る）を強いるもの＝難しい」**。失敗モード5つ（機械的消去／鍵の提示／反復で飽きる／最終が推測可能／法則が恣意的）を設計時に潰す（§6.4）。
- **物語接地**：謎の法則・解錠条件・組み合わせ規則は**世界の理から必然として立ち上がる**こと。恣意的なゲーム都合にしない（§6.2。反例 `apothecary_brew` ↔ 正例 `legendary_forge_cmd`）。
- 引き出しは連想／証拠解釈／論理／嘘つき／アリバイ／水平思考／換字／**帰納**／**帰納×演繹の融合** など（§4・§6.1）。

### 2. シナリオの実装

Ink 構文で実装します。基本は [ink_basics.md](./references/ink_basics.md)、複雑なロジックは [ink_advanced.md](./references/ink_advanced.md)。

- **固定コマンドパレットで実演させる**：謎は独自トグルUIや「答えを選ぶ」N択でなく、**固定の動詞（しらべる→対象／つかう→道具→対象／もちもの 等）＋ outcome-gating** で実演させる（`PUZZLE_DESIGN.md` §0・§6、PoC: `escape_room.ink` / `legendary_forge_cmd.ink`）。解決専用コマンドを湧かせない。N択コミットは結論の「最後の確認」に留める。
- **公開ステータス変数 `public_status` を必ず宣言**：AI に見せてよいステータス（HP・所持金・現在地など）の変数名をカンマ区切りで列挙。解法・真相フラグ（`has_*`/`is_*`/`knows_*`/正解値）は載せない。書式・判断基準は [adventure_mcp_patterns.md](./references/adventure_mcp_patterns.md) の該当節を参照。

### 3. ファイルの出力

実装したシナリオは `engine/assets/` 配下に `.ink` で保存します。

### 4. コンパイルと一意性の機械検証

- `pnpm run build:ink` で JSON にコンパイルできるか確認（エラーは標準出力をもとに修正）。
- **論理パズルは唯一解であることを別途スクリプトで機械検証**してから先へ進む（全解空間を総当たりし、解がちょうど1つであること／意図した反例・罠が仕様どおりであることを確認）。融合型は「演繹のみ=N候補／法則込み=一意」を検証する。**これを怠ると複数解・解なしのロジックバグが残る。**

### 5. スモークプレイ／コールドプレイ（必須）

`build:ink` の成功だけでは実行時の破綻（ソフトロック・一度きりイベント再発・解けない謎）を検出できない。**生成 JSON を inkjs で再生**して必須項目を確認する（先頭 BOM を `replace(/^﻿/, "")` で除去）。新シナリオ（`engine/assets/` のライブ作品）には必ずスモークプレイテストを添える（過去の参照シナリオ集は `docs/06-reference/scenarios/`）。最低限：

- デッドエンド（ソフトロック）が存在しない／クリア（`-> DONE`/`-> END`）へ到達可能
- 素朴に進む失敗ルートが存在し挽回できる（致死/バッドは終端EDでソフトロックでない）
- 一度きりイベントが再訪で再発しない
- 前提ゲート型戦闘は「必須装備なしでは勝てない／ありでは勝てる」を両方検証

「面白さ・難易度・フェアさ」は決定的テストでは測れない。必要に応じ**コールドプレイ**（解を見ていない別エージェントが本文だけで実プレイし、論理で解けるか／総当たり耐性／理不尽でないかを報告）を品質シグナルとして使う。スモークプレイ結果から難易度を推定し、ヒアリングした方針との乖離があれば調整提案する。

### 6. 全自動（おまかせ）の整合チェック関所（人間の一読）

AI は自分の物語の破綻・難易度のズレを自己検知しにくい。おまかせモードでは**完成時に、解を伏せたまま「前提・動機・"なぜその仕掛けが世界で成立するか"・構成・推定難易度・冒頭の本文サンプル」をユーザーに提示**し、方向性ズレ（テーマ解釈違い・難易度ズレ・物語の破綻）を人間が一読で弾けるようにする。リッチで信じられる物語が要る場合は**共作モードへ誘導**する（おまかせは「薄いが内部整合した枠」に留め、リッチで破綻した物語を盲目で書こうとしない）。

### 7. 同期（rulesync 源泉を編集した場合）

スキル本体や reference を編集したら `npx rulesync generate` を実行し、`.claude/` `.agents/` の生成物が源泉と一致すること（`git diff` で差分なし）を確認する。`.claude/` `.agents/` を手で編集しない。
