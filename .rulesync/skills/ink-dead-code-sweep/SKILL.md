# Ink Dead-code Sweep

Ink シナリオの**死蔵コード整理**専用スキル。到達不能 knot・未使用 function・未使用/書き込み専用 VAR を
機械検出し、**外部参照が無いことを確かめてから**削除し、**挙動を変えていない**ことを確認して終える。

- 作成は `ink-scenario-creator`、作法レビューは `longform-scenario-review`、**整理はこのスキル**。
- レビューが「指摘まで」なのに対し、本スキルは**安全に削除まで**行う。ただし削除は機械的な candidate を
  鵜呑みにせず、§3 の確認を必ず通す。

## 手順

### 1. 検出（スクリプト）

```
node .rulesync/skills/ink-dead-code-sweep/scripts/scan_ink.mjs <path/to/scenario.ink>
```

`// … ` と `/* … */` を除去してから解析し、次を列挙する（`名前 (file:line)` 付き）：

- **到達不能な knot** … 定義はあるが `-> 名前` / `<- 名前` の的に一度も現れない（function・エントリ knot・
  `DONE`/`END`/`START` は除外）。
- **未使用の function** … 定義はあるが `名前(` の呼び出しが本体以外に無い。
- **未使用 VAR：宣言のみ** … 宣言だけで読みも書きも無い。
- **未使用 VAR：書き込み専用** … `~ 名前 = …` の代入はあるが「読み」が 0（死蔵フラグ）。
- **予約変数** … `public_status`。read=0 でも**消さない**（後述）。

> Windows では Git Bash の `/tmp` を Node が解釈できない。一時ファイルはリポジトリ相対パスか
> scratchpad（Windows パス）に置いて渡すこと。

### 2. candidate であって確定ではない、と心得る

スクリプトは静的解析なので次は検出しきれない。**消す前に必ず人間が確認**する：

- divert 先が**変数経由**（divert 型変数）や `function` の戻り値で決まる動的分岐。
- knot が**外部（MCP/テストのジャンプ）**から名指しで開始される場合。
- Ink の knot は**フォールスルーしない**前提で「到達不能」を出している。stitch/gather の内部到達は別。

### 3. 削除可否の確認（最重要）

「未使用 VAR：書き込み専用／宣言のみ」でも、**Ink の外側が読んでいれば消せない**。削除前に必ず：

1. **予約変数 `public_status` は残す**。エンジンが公開ステータスとして読む（`cli/.../game-controller.ts`）。
   値に列挙された変数（例 `"place"`）も公開対象なので消さない。
2. **テスト／エンジンの外部参照を grep**：
   ```
   grep -rn 'getVariable("名前")' --include='*.ts' .   # テストが値を読む
   grep -rn '\b名前\b' --include='*.ts' --include='*.json' . | grep -v node_modules
   ```
   `engine/src/**/**.spec.ts` がその変数を `getVariable` で読むなら、それは「テストだけで使う変数」。
   ユーザーがテスト専用変数の削除を許可しているか確認する（既定では**消さない**＝テストが赤くなる）。
3. **set-only フラグの意図**：進行マーカー（`departed*` 等）や演出フラグが「将来の配線用に意図的に
   置かれている」場合がある。読みが 0 でも、消してよいかは作者に一言確認するのが安全。

### 4. 削除と巻き添えの掃除

- knot を消すなら、その knot 内でしか書かれない VAR の宣言も一緒に消す（逆も同様）。
- divert を消すなら、孤立した分岐選択肢・到達不能化する後続 knot を連鎖的に見直す。
- 消した識別子を**コメントが指していないか**確認し、古くなった設計コメントも実態に合わせて直す。
- 整理後にもう一度スクリプトを回し、**candidate が 0（予約変数のみ）**になったことを確認する。

### 5. 挙動が変わっていないことの確認

死蔵削除は**振る舞いを変えない**のが原則。次で担保する：

1. **再コンパイル**：`cd engine && node scripts/build-ink.ts`（`.json` を再生成）。エラーが出れば
   宙に浮いた divert や未定義参照が残っている合図。
2. **テスト**：`cd engine && npx vitest run scenarios-structure`（と関連 spec）。
3. **回帰の切り分け**：もともと赤い WIP テストがある場合に備え、**削除前後で pass/fail 数が同一**で
   あることを基準にする（例：コミット版を `git stash`/`git checkout -- 対象` で一旦戻してビルド＆テスト→
   ベースラインの pass/fail を控え、自分の版に戻して同数を確認）。新たな赤が増えていなければ behavior-neutral。

### 6. 報告

何を（到達不能 knot／未使用 function／未使用 VAR）どれだけ消したか、**残した予約変数・外部参照のあるもの**、
ビルド・テスト結果（ベースラインと同数である旨）を、`file:line` 付きで簡潔にまとめる。コミットは指示があるまで行わない。

## 落とし穴メモ

- `public_status` はリネーム不可の予約変数。read=0 でも常に残す。
- `~ a = a + 1` のような自己参照代入は **RHS が読み**。スクリプトはこれを読みとして数える（カウンタは死蔵にしない）。
- 「テストだけが `getVariable` する変数」はシナリオ本体からは未使用に見える。削除はユーザー合意のうえで。
- コメント内の名前は読みに数えない（スクリプトはコメント除去後に解析）。設計メモが古くなりがちなので、
  削除時にコメントも追従させる。
