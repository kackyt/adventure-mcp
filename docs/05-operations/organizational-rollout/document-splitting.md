# ドキュメント分割の手順 (Document Splitting)

> **適用範囲**: 単一の Markdown 文書が肥大化したときの分割判断と機械的手順。コア 7 文書および拡張文書すべてに適用する。

## なぜ分割するのか

- **AI ツールの context window** には実質的な上限があり、巨大な単一文書は読み込み時にトークンを浪費する。
- **人間のレビュアー** にとっても、500 行を超えた変更は「変更箇所の意図」を見失いやすい。
- 文書が小さく分割されていれば、**変更影響範囲が局所化** され、Git の diff も読みやすい。

## 分割閾値（書籍 第14章準拠）

| 行数      | 判断           | 推奨アクション                             |
| --------- | -------------- | ------------------------------------------ |
| 〜 500 行 | 適正           | そのまま運用                               |
| 500 行超  | **分割を検討** | 章単位で重複・冗長がないか棚卸し           |
| 800 行超  | **分割を推奨** | 親（索引）+ 子（詳細）構造に再編           |
| 1200 行超 | **分割を必須** | リリース前に必ず分割。レビュー前提が崩れる |

> 既存テンプレートの一部には旧基準（800/1200/1500 行）の記述が残る場合がある。**新規文書および分割判断は本ドキュメントの値（500/800/1200）を優先** する。

### 閾値の根拠

- **500 行**: AI ツールが 1 ファイルを「1 chunk」として安定読込できる目安。
- **800 行**: 人間が 1 セッションで通読できる上限。これを超えるとレビュー疲労が増える。
- **1200 行**: diff レビューが現実的に困難になる線。1200 行超の単一 PR 変更は構造的に問題。

## 分割の構造（親 + 子）

```text
親ドキュメント（索引）              # 200-300 行
├── サブディレクトリ/
│   ├── トピック1.md                # 300-800 行
│   ├── トピック2.md                # 300-800 行
│   └── トピック3.md                # 300-800 行
```

**親の役割**:

- 全体像（システム図・目次・推奨読み順）の提示
- 子文書へのリンク集
- クイックスタート（30 秒で全体像をつかめるサマリ）

**子の役割**:

- 1 トピックを **300〜800 行** で完結させる
- 親には書かない詳細（手順・コード例・落とし穴）

### 実例（本リポジトリ）

`docs-template/05-operations/DEPLOYMENT.md` は元々 2,000 行超の単一文書だったが、現在は **索引 + サブ文書群** に分割済み（最新の構成は `docs-template/05-operations/DEPLOYMENT.md` および `deployment/` ディレクトリを参照）。

```text
DEPLOYMENT.md（索引、300 行未満を維持）
└── deployment/
    ├── git-workflow.md            # AI 駆動 Git Workflow
    ├── self-review.md             # セルフレビュー詳細
    ├── knowledge-management.md    # ナレッジ体系化
    ├── ai-tools-integration.md    # AI ツール統合
    ├── ci-cd.md                   # CI/CD パイプライン
    ├── infrastructure.md          # インフラ・ロールバック
    ├── monitoring.md              # モニタリング・運用
    └── ...（他複数）
```

> 上記は構造例。具体ファイル数・各サイズは継続的に変動するため、最新は `ls docs-template/05-operations/deployment/` または [GitHub の `deployment/` ディレクトリ](https://github.com/feel-flow/ai-spec-driven-development/tree/develop/docs-template/05-operations/deployment) で確認してください。

## 分割の標準手順（チェックリスト）

### Step 1: 棚卸し（30 分）

- [ ] 元文書の **見出し（H2/H3）一覧** を抽出する。
- [ ] 各セクションの **行数** を計測する（`awk` や VS Code の outline で可）。
- [ ] **依存グラフ** を書き出す: 「A 章は B 章を前提にしているか」。

### Step 2: 子トピックの粒度決定

- [ ] 1 子文書 = **1 概念 / 1 ワークフロー / 1 ガイド** の単位に揃える。
- [ ] 想定読者の **入口**（最初に読む人）と **退路**（次に読むべき文書）を各子で 1 行記載できるか。
- [ ] 子文書名は **動詞 or 名詞句** で簡潔に（例: `git-workflow.md` / `self-review.md`）。

### Step 3: 親（索引）の再構築

- [ ] 親に残すのは **目次・全体像・クイックスタート** のみ。
- [ ] 親が **300 行以内** に収まることを確認。
- [ ] 子へのリンク表に「推奨読み順」「想定読者」を併記。

### Step 4: 移行・リンク張替え

- [ ] 親→子のリンクを相対パスで張る (`./subdir/topic.md`)。
- [ ] 元文書を参照していた **他の文書からのリンク** を全件更新。
  - `grep -rln "OLD_DOC.md" docs-template/ docs/` で機械的に検索。
- [ ] **MASTER.md / DECISION_MATRIX.md / GETTING_STARTED.md** からの参照を最優先で更新。

### Step 5: 検証

- [ ] `npm run lint:md` でリンク切れ・heading 構造を確認。
- [ ] `npm run validate` でコア 7 文書の存在チェックを通す（任意ディレクトリを検証する場合のみ `-- <dir>` で渡す）。
- [ ] AI ツールに「○○について教えて」と問い、親→子の経路で正しい回答が得られるか確認。

## 分割しないほうが良いケース

| 状況                                      | 判断                                                                 |
| ----------------------------------------- | -------------------------------------------------------------------- |
| 1200 行超だがほぼ表データ（マトリクス等） | 分割しない。表は途中で切ると読めない                                 |
| 1200 行超だが生成物（OpenAPI 等）         | 分割しない。手で書いていない文書は対象外                             |
| 600 行で頻繁に diff 発生                  | **分割を推奨**。閾値以下でも変更頻度が高い場合は子化する             |
| 400 行で 6 トピックが並列                 | **構造リファクタを検討**。閾値未達でも責務が混在しているなら整理する |

## 親文書に書きがちな落とし穴

| アンチパターン                           | 対策                                                                |
| ---------------------------------------- | ------------------------------------------------------------------- |
| 親に詳細手順を全部書いてしまう           | 親は索引。詳細は子に移し、親には **3 行以内** のサマリだけ残す      |
| 子のコピペが親に残ったまま               | DRY 違反。親と子で重複したら **子を SSOT** とし、親は参照リンクのみ |
| 子文書が単独で読めない                   | 子の冒頭に「**前提**」「**この文書の範囲**」を 1 段落で明記する     |
| サブディレクトリ名が抽象的（`misc/` 等） | サブディレクトリ名は **親文書名と対応** させる（例: `deployment/`） |

## 既存文書の事後分割の進め方

1. **ブランチを切る**: `chore/#NNN-split-OLD_DOC`
2. **1 PR で 1 文書のみ** 分割（複数まとめると diff 爆発）
3. **`git mv` で履歴を保つ**: 子文書化する際は `git mv` してから内容を再構成
4. **段階的にリンクを更新**: 1 PR で完結させるが、コミットは「split」「relink」「polish」の 3 段に分けると読みやすい
