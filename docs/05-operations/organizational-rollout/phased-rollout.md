# 段階的導入ガイド (Phased Rollout)

> **適用範囲**: 既存組織・チームへ AI 仕様駆動開発フレームワークを段階導入する際の手順。新規プロジェクト立ち上げの場合は [GETTING_STARTED_NEW_PROJECT.md](../../GETTING_STARTED_NEW_PROJECT.md) を併読する。

## なぜ段階導入か

書籍 第15章「チーム標準化」で示されている通り、ドキュメント駆動・AI 駆動の運用は **一括導入すると拒否反応を起こしやすい**。Phase を区切り、各段階で **明確に体感できる成果（quick win）** を提示することで、自発的な定着を促す。

| Phase   | 期間目安 | 範囲                     | Quick win の例                         |
| ------- | -------- | ------------------------ | -------------------------------------- |
| Phase 1 | 1 週間   | MASTER.md のみ           | AI ツールの応答ブレが減る              |
| Phase 2 | 2 週間   | + Issue テンプレート     | 着手前の認識ズレが減る                 |
| Phase 3 | 1 ヶ月   | + 残り 6 文書            | 横断レビュー・新人 onboarding が高速化 |
| Phase 4 | 1 ヶ月   | + 自動チェック (CI/Hook) | レビュー指摘が「人間判断」に集中できる |

> 期間はあくまで目安。チーム規模・既存ドキュメント量によって 1.5〜2 倍まで揺れる前提で計画する。

## Phase 1: MASTER.md のみ導入（1 週間）

### 目的

- AI ツール（Claude Code / GitHub Copilot / Cursor）の応答品質を「中央集権ファイル 1 つ」で底上げする。
- 既存ワークフローを **一切変えない** ことで導入摩擦をゼロに近づける。

### 作業内容

1. **テンプレートをコピー**: `docs-template/MASTER.md` をプロジェクト直下にコピー（`docs/MASTER.md` でも可）。
2. **必須情報のみ埋める**:
   - プロジェクト名・ターゲットユーザー・主要機能（5 行以内で十分）
   - 採用済み技術スタック（言語・FW・DB）
   - 既知の禁止事項（Magic number 禁止 等の明確なルールがあれば追記）
3. **AI ツールに参照させる**: `CLAUDE.md` / `.github/copilot-instructions.md` / `.cursor/rules` のいずれかに `MASTER.md を最初に読むこと` を 1 行追加。

### 完了判定（Definition of Done）

- [ ] AI に「このプロジェクトの技術スタックは？」と聞いて、MASTER.md の値を返す。
- [ ] チーム内 1 名以上が「AI の応答が以前より具体的になった」と体感報告。

### この Phase で **やらないこと**

- PROJECT/ARCHITECTURE 等の他 6 文書には触れない（負荷を上げない）。
- Issue テンプレート変更しない。
- 自動 lint・CI 設定しない。

## Phase 2: Issue テンプレートの追加（2 週間）

### 目的

- 「何をやるか」をチケット時点で構造化し、AI に渡すコンテキストを整える。
- 着手前の **認識ズレと手戻り** を減らす。

### 作業内容

1. **Issue テンプレート 4 種を導入**:
   - `feature.md` / `bug.md` / `chore.md` / `docs.md` の 4 種。
   - サンプルは `.github/ISSUE_TEMPLATE/` 配下のテンプレートを流用（本リポジトリのテンプレートを参照）。
2. **「スコープ確認」チェックボックスを必須化**: 1 Issue = 1 成果物の原則。
3. **AI 駆動 Refine の運用開始**: `/refine-issue` 相当の習慣を導入し、曖昧仕様を着手前に潰す。

### 完了判定

- [ ] 直近 2 週間に作成された Issue の 80% 以上がテンプレートに沿っている。
- [ ] 「Issue が曖昧で着手後に質問した」事例が前期間より減少（簡易アンケートで確認）。

### この Phase で **やらないこと**

- 残り 6 文書はまだ書かない（次 Phase へ）。
- PR テンプレート整備は任意（負荷次第）。

## Phase 3: 残り 6 文書の段階追加（1 ヶ月）

### 目的

- AI が「設計判断・ドメイン語彙・実装規約・テスト方針・運用」を一気通貫で把握できる状態を作る。
- 6 文書を **同時に書こうとしない** こと。優先順位に従って 1 文書ずつ追加する。

### 推奨追加順

| 順序 | 文書                            | 理由                                                            |
| ---- | ------------------------------- | --------------------------------------------------------------- |
| 1    | `01-context/PROJECT.md`         | 全文書のビジネス前提を提供する根本                              |
| 2    | `03-implementation/PATTERNS.md` | 日々の実装に最も影響する。コーディング規約・anti-pattern の集約 |
| 3    | `02-design/ARCHITECTURE.md`     | システム俯瞰。レビュー・新規機能設計の参照頻度が高い            |
| 4    | `02-design/DOMAIN.md`           | ドメインモデル・ユビキタス言語。AI の用語ブレを抑える           |
| 5    | `04-quality/TESTING.md`         | テスト戦略。CI 整備（Phase 4）の前提資料になる                  |
| 6    | `05-operations/DEPLOYMENT.md`   | 本番運用フローが固まってから着手するほうが手戻りが少ない        |

### 各文書の最小要件

- **300 行以内** で書き始める。網羅より **頻出シーン × 1 例** を優先。
- 1 文書 1 週間。次の文書に着手する前にチームレビュー（30 分以内）を 1 回挟む。

### 完了判定

- [ ] コア 7 文書すべてに最低限のコンテンツ（実プロジェクト固有の情報）が入っている。
- [ ] AI に「PR レビュー観点は？」と聞くと、PATTERNS.md / TESTING.md 由来の具体観点を返す。

### この Phase で **やらないこと**

- 完璧主義に陥らない。**80% で次へ**。
- ドキュメント分割の閾値（500 行）に達した文書だけ [document-splitting.md](./document-splitting.md) を参照する。

## Phase 4: 自動チェック導入（1 ヶ月）

### 目的

- 人間レビュアーが **本質的な判断** に集中できるよう、機械的に検出可能な逸脱を自動化する。

### 作業内容（優先順）

1. **markdownlint + prettier**:
   - 文書の lint（リンク切れ・heading 構造・行末）を CI/Hook で強制。
2. **コア 7 文書存在チェック**:
   - `scripts/validate-docs.mjs` 相当。MASTER.md からの参照リンク切れを検出。
3. **Issue/PR テンプレート遵守チェック**:
   - GitHub Action もしくは Husky で「スコープ確認」チェックボックスの未記入を弾く。
4. **AI 駆動レビュー**:
   - `/pr-review-toolkit:review-pr` 等のセルフレビュー導入（PR 着手前にローカル実行）。
5. **PLAYBOOK 更新の自動 lint**:
   - 知見抽出（ACE）後の差分が PLAYBOOK のカテゴリサイズ規約を超えないか機械チェック。

### 完了判定

- [ ] PR 作成時、文書 lint と存在チェックが自動実行される。
- [ ] 直近 4 週間で「人間レビュアーの指摘の 50% 以上が _機械では検出できない_ 設計判断」になっている。

### この Phase で **やらないこと**

- 完全自動化を目指さない。AI レビューは **人間の補助** として運用する。

## 全 Phase 共通の落とし穴

| アンチパターン                            | 対策                                                                                      |
| ----------------------------------------- | ----------------------------------------------------------------------------------------- |
| 「最初から全部揃えよう」                  | Phase を飛ばすと運用負荷で頓挫する。1 Phase = 1 quick win に絞る                          |
| 「ドキュメントは完璧にしてから公開」      | 80% で公開し、運用しながら直す。**未完 = 公開しない** は禁止                              |
| 「AI ツールの設定だけ整えて文書は後回し」 | AI の精度は文書品質に従属する。設定だけ先行しても効果が出ない                             |
| 「Phase 4 の自動化を Phase 1 で始める」   | 文書がないのに lint 強化しても弾くものがない。Phase 4 は Phase 3 完了後                   |
| 「分割しないまま 1200 行超」              | [document-splitting.md](./document-splitting.md) の閾値（500/800/1200）に従って早めに分割 |
