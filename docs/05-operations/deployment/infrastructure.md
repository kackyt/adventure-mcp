# Infrastructure and Deployment Operations

> **Parent**: [DEPLOYMENT.md](../DEPLOYMENT.md)

本ドキュメントはインフラストラクチャ管理、デプロイメント自動化、環境設定、ロールバック戦略、災害復旧に関する運用ガイドです。

## 1. Infrastructure as Code (IaC)

### Terraform構成概要

```
infrastructure/
├── main.tf                 # メイン設定
├── variables.tf            # 変数定義
├── outputs.tf              # 出力値
└── modules/
    ├── vpc/                # ネットワーク
    ├── ecs/                # コンテナサービス
    ├── rds/                # データベース
    └── monitoring/         # 監視設定
```

### コアリソース構成

**VPC設定**:

- CIDR: `10.0.0.0/16`
- マルチAZ構成（ap-northeast-1a, 1c）
- パブリック/プライベートサブネット分離

**ECS Cluster**:

- Container Insights有効化
- Fargate起動タイプ
- Auto Scaling対応（min: 2, max: 10）

**デプロイメント設定**:

- Rolling Update戦略
- `maximum_percent`: 200%
- `minimum_healthy_percent`: 100%

**Auto Scaling基準**:

- CPU使用率: 70%でスケールアウト
- メトリクス収集: 1分間隔
- クールダウン期間: 300秒

## 2. デプロイメント自動化

### Blue-Greenデプロイメント手順

**フェーズ1: 準備**

1. 新バージョンのタスク定義登録
2. 新ターゲットグループ作成（Green）
3. ヘルスチェックパス設定: `/health`

**フェーズ2: デプロイ**

1. Greenサービスを起動（desired_count: 4）
2. ヘルスチェック待機（60秒）
3. 全コンテナの健全性確認

**フェーズ3: 切替**

1. ALBリスナーをGreenに向ける
2. トラフィック完全移行確認
3. 5分間の安定性監視

**フェーズ4: クリーンアップ**

1. Blueサービスのdesired_count → 0
2. 旧ターゲットグループ削除（24時間後）
3. デプロイメントログ記録

### デプロイメントスクリプト構造

```bash
scripts/
├── deploy-production.sh     # 本番デプロイ
├── deploy-staging.sh        # ステージングデプロイ
├── rollback.sh              # ロールバック
└── health-check.sh          # ヘルスチェック
```

**必須パラメータ**:

- `VERSION`: デプロイバージョン（セマンティックバージョニング）
- `CLUSTER`: ECSクラスタ名
- `SERVICE`: サービス名
- `TASK_DEFINITION`: タスク定義ファミリー名

**エラーハンドリング**:

- `set -e`: エラー時即座に終了
- 各ステップでの健全性確認
- 失敗時の自動通知（Slack/Email）

## 3. 環境設定管理

### 環境変数体系

**階層構造**:

```
config/
├── index.ts              # 設定エントリポイント
├── environments/
│   ├── development.ts    # 開発環境
│   ├── staging.ts        # ステージング環境
│   └── production.ts     # 本番環境
└── validation.ts         # 設定検証ロジック
```

**設定カテゴリ**:

1. **アプリケーション設定**
   - `PORT`: サーバーポート（デフォルト: 3000）
   - `NODE_ENV`: 実行環境（development/staging/production）
   - `APP_NAME`: アプリケーション識別名

2. **データベース設定**
   - `DB_HOST`, `DB_PORT`: 接続情報
   - `DB_NAME`, `DB_USER`, `DB_PASSWORD`: 認証情報
   - 本番環境では必須検証

3. **キャッシュ設定**
   - `REDIS_HOST`, `REDIS_PORT`: Redis接続情報

4. **AWS設定**
   - `AWS_REGION`: リージョン（デフォルト: ap-northeast-1）
   - `S3_BUCKET`: ストレージバケット名

### Secrets管理戦略

**保存場所**:

- AWS Secrets Manager（本番/ステージング）
- `.env.local`（開発環境のみ）
- Kubernetes Secrets（K8s環境）

**管理対象**:

```yaml
app-secrets:
  - database-password # DB認証情報
  - jwt-secret # JWT署名キー
  - api-keys # 外部API認証キー
  - encryption-keys # データ暗号化キー
```

**ローテーション**:

- データベースパスワード: 90日ごと
- APIキー: 180日ごと
- JWT秘密鍵: 年1回

**アクセス制御**:

- IAMロールベース認証
- 最小権限の原則
- アクセスログ監査

## 4. ロールバック戦略

### 自動ロールバック条件

**監視メトリクス**:

| メトリクス          | 閾値     | 評価期間 | アクション                          |
| ------------------- | -------- | -------- | ----------------------------------- |
| エラー率            | > 5%     | 5分      | 即座にロールバック                  |
| P99レスポンスタイム | > 1000ms | 3分      | 警告 → 10分継続でロールバック       |
| メモリ使用率        | > 90%    | 2分      | スケールアウト → 失敗時ロールバック |
| CPU使用率           | > 85%    | 5分      | スケールアウト試行                  |
| ヘルスチェック失敗  | > 50%    | 1分      | 即座にロールバック                  |

### ロールバック実行フロー

**Phase 1: 検出**

1. メトリクス異常検知
2. アラート発火
3. 自動評価開始（30秒以内）

**Phase 2: 判定**

1. 複数メトリクスの相関分析
2. ロールバック必要性判定
3. ステークホルダー通知

**Phase 3: 実行**

1. 前バージョンのタスク定義取得
2. ECSサービス更新（前バージョンへ）
3. トラフィック切替確認

**Phase 4: 検証**

1. ロールバック後のヘルスチェック
2. メトリクス正常化確認
3. インシデントレポート作成

### 手動ロールバック

**コマンド実行**:

```bash
./scripts/rollback.sh <deployment-id> [--dry-run]
```

**確認項目**:

- 前バージョンの特定
- データベース互換性
- 設定変更の巻き戻し必要性
- ユーザーへの影響範囲

## 5. 災害復旧 (DR)

### バックアップポリシー

| 対象                   | 頻度   | 保持期間 | 方式                        | リカバリ目標                 |
| ---------------------- | ------ | -------- | --------------------------- | ---------------------------- |
| データベース           | 日次   | 30日     | 自動スナップショット + PITR | RPO: 5分, RTO: 1時間         |
| アプリケーションデータ | 時間   | 7日      | S3増分バックアップ          | RPO: 1時間, RTO: 30分        |
| 設定ファイル           | 変更時 | 90日     | Git + S3バージョニング      | RPO: 0, RTO: 15分            |
| ログデータ             | 継続的 | 90日     | CloudWatch Logs             | RPO: リアルタイム, RTO: 即座 |

**PITR (Point-in-Time Recovery)**: 任意の時点への復旧が可能

### 復旧手順

**Level 1: サービス部分障害**

1. 異常コンテナの自動再起動
2. Auto Scalingによる補完
3. 所要時間: 5分以内

**Level 2: データベース障害**

1. 最新スナップショット特定
2. 新インスタンスへの復元
3. アプリケーション接続先変更
4. 所要時間: 30分-1時間

**Level 3: リージョン全体障害**

1. 別リージョンでのインフラ起動
2. データベースのクロスリージョンレプリカ昇格
3. Route53でのDNSフェイルオーバー
4. 所要時間: 2-4時間

### 災害復旧スクリプト

**データベース復旧**:

```bash
# 最新スナップショットから復元
aws rds restore-db-instance-from-db-snapshot \
  --db-instance-identifier restored-db \
  --db-snapshot-identifier <snapshot-id>
```

**アプリケーション復旧**:

```bash
# 災害復旧モードでデプロイ
./scripts/deploy-production.sh disaster-recovery
```

**DNSフェイルオーバー**:

```bash
# Route53レコード変更
aws route53 change-resource-record-sets \
  --hosted-zone-id <zone-id> \
  --change-batch file://dns-failover.json
```

### 復旧訓練

**頻度**: 四半期ごと

**訓練シナリオ**:

1. データベース完全消失
2. リージョン障害
3. データ破損からの復旧
4. セキュリティインシデント対応

**検証項目**:

- RTO/RPO目標の達成
- 手順書の正確性
- チーム連携の円滑性
- 通知システムの動作

## 6. デプロイメントチェックリスト

### Pre-Deployment（デプロイ前）

**コード品質**:

- [ ] すべてのユニットテストがパス（カバレッジ ≥ 80%）
- [ ] E2Eテストが成功
- [ ] Lintエラーなし
- [ ] セキュリティスキャン完了（脆弱性: Critical 0件, High 0件）

**レビュー**:

- [ ] コードレビュー完了（最低2名の承認）
- [ ] アーキテクチャ変更の承認（必要に応じて）
- [ ] セキュリティレビュー完了（機密情報扱う変更の場合）

**インフラ準備**:

- [ ] データベースマイグレーション計画確認
- [ ] ロールバック計画文書化
- [ ] リソース容量確認（CPU/メモリ/ストレージ）

**コミュニケーション**:

- [ ] デプロイメント通知送信（#deployments Slackチャネル）
- [ ] メンテナンス窓の設定（必要に応じて）
- [ ] オンコール担当者の確認

### During Deployment（デプロイ中）

**リアルタイム監視**:

- [ ] デプロイメントログを監視
- [ ] エラー率メトリクス監視（< 1%）
- [ ] レスポンスタイム監視（P99 < 500ms）
- [ ] リソース使用率監視（CPU < 70%, Memory < 80%）
- [ ] ヘルスチェックステータス確認

**段階的確認**:

- [ ] タスク定義登録成功
- [ ] 新コンテナ起動成功
- [ ] ヘルスチェック通過
- [ ] トラフィック切替完了

### Post-Deployment（デプロイ後）

**動作確認**:

- [ ] スモークテスト実行（主要エンドポイント）
- [ ] 主要機能の手動確認
- [ ] データベース接続確認
- [ ] 外部API連携確認

**メトリクス検証**:

- [ ] エラーログ確認（15分間）
- [ ] パフォーマンスメトリクス確認（30分間）
- [ ] ユーザーフィードバック監視
- [ ] ビジネスメトリクス確認（重要KPI）

**記録**:

- [ ] デプロイメント完了報告
- [ ] インシデント記録（発生時）
- [ ] 改善点のドキュメント化
- [ ] 次回デプロイへの引継ぎ事項記載

## 7. モニタリング設定

### CloudWatch Alarms

**アラーム定義**:

| アラーム名          | メトリクス          | 閾値         | 評価期間 | アクション       |
| ------------------- | ------------------- | ------------ | -------- | ---------------- |
| HighCPUUtilization  | CPUUtilization      | 80%          | 2期間    | スケールアウト   |
| HighErrorRate       | HTTPCode_5XX        | 10件/分      | 1期間    | 即座に通知       |
| LowHealthyHosts     | HealthyHostCount    | < 2          | 1期間    | クリティカル通知 |
| HighMemoryUsage     | MemoryUtilization   | 85%          | 2期間    | スケールアウト   |
| DatabaseConnections | DatabaseConnections | > 80% of max | 1期間    | 警告通知         |

**通知チャネル**:

- Slack: `#alerts` チャネル
- Email: on-call担当者
- PagerDuty: クリティカルアラートのみ
