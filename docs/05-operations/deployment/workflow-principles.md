# ワークフロー運用原則

> **Parent**: [DEPLOYMENT.md](../DEPLOYMENT.md) | **Related**: [git-workflow.md](./git-workflow.md)

## 概要

AI駆動Git Workflowを効率的に運用するための基本原則を定義します。これらの原則はワークフロー全体に適用され、不要な中断を排除し、スコープの肥大化を防ぎます。

---

## 3つの運用原則

### 原則1: ノンストップフロー

**ルール**: Issue作成からPR作成まで、不要な確認を挟まず一気通貫で進める

```
Issue作成 → ブランチ作成 → 実装 → テスト → セルフレビュー → PR作成
                    ↑ この一連の流れを中断しない ↑
```

**やること**:

- 各ステップを順序通りに連続実行
- セルフレビューの指摘は即座に修正してコミット
- レビュー対応ポリシーに従い、Critical/Warning は確認不要で即対応

**やらないこと**:

- 途中で不要な「これでいいですか？」確認を挟む
- 実装完了後にPR作成を後回しにする
- セルフレビュー結果を報告して承認を待つ

---

### 原則2: スコープ外発見のIssue化

**ルール**: 作業中にスコープ外の改善点・バグを見つけた場合、その場で修正せず新規Issueを作成する

**やること**:

- スコープ外の発見を新規Issueとして記録
- 現在のタスクに集中し続ける
- Issue作成時に発見の経緯を簡潔に記載

**やらないこと**:

- 「ついでに」修正する
- PRのスコープを拡大する
- 発見を無視して記録しない

**例**:

```bash
# 作業中にリファクタリング機会を発見した場合
gh issue create \
  --title "refactor: UserService のメソッド分割" \
  --body "## 発見の経緯
Issue #123 の作業中に発見。UserService.createUser() が150行を超えており、
バリデーション・永続化・通知の責務が混在している。

## 提案
- バリデーション、永続化、通知を個別メソッドに分離
- 単体テストの追加

## 優先度
低（機能的には正常に動作中）" \
  --label "refactoring"
```

---

### 原則3: 曖昧仕様の確認タイミング

**ルール**: 要件が不明確な場合はIssue着手前にユーザーに確認する。作業開始後に出てきた疑問は別Issue化して別対応とし、現在の作業を止めない

| タイミング                   | 対応                                          |
| ---------------------------- | --------------------------------------------- |
| Issue着手前                  | 曖昧な要件・仕様をユーザーに確認              |
| 作業開始後（軽微な疑問）     | 別Issueを作成して別対応。現在の作業は止めない |
| 作業開始後（致命的な不明点） | 例外的に作業を止めてユーザーに確認してよい    |

**致命的な不明点の判断基準**:

- 確認しないと実装が物理的に進められない
- 誤った実装をすると大幅な手戻りが発生する
- セキュリティやデータ整合性に直結する

## タスク管理: TodoWrite

ワークフローの各ステップを TodoWrite で管理し、進捗を可視化します。

### 標準チェックリストテンプレート

ワークフロー開始時に以下のタスクリストを作成：

```
1. [ ] GitHub Issue 作成
2. [ ] feature ブランチ作成
3. [ ] 実装
4. [ ] テスト実行・合格確認
5. [ ] セルフレビュー: PR Review Toolkit
6. [ ] セルフレビュー: Codex CLI クロスモデルレビュー
7. [ ] レビュー指摘修正・コミット
8. [ ] Push + PR 作成
9. [ ] (レビュー指摘対応)
10. [ ] (レビュワーへのサマリーコメント)
```

### 運用ルール

- タスク開始時に `in_progress` に更新
- タスク完了時に即座に `completed` に更新
- **同時に `in_progress` のタスクは1つだけ**
- スコープ外の発見は新規タスクとして追加せず、Issue化（原則2）

---

## 原則の適用タイミング

| ワークフローステップ | 適用される原則                                                |
| -------------------- | ------------------------------------------------------------- |
| Issue作成            | 原則3（曖昧仕様の確認）                                       |
| ブランチ作成〜PR作成 | 原則1（ノンストップフロー）                                   |
| 実装中の発見         | 原則2（スコープ外Issue化）                                    |
| セルフレビュー       | 原則1 + [Review Response Policy](./review-response-policy.md) |
| レビュー対応         | 原則1 + 原則2                                                 |
