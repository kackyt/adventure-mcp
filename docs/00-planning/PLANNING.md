# プロジェクト企画書テンプレート

## 📋 このドキュメントの目的

このテンプレートは、**何も決まっていない状態から、AIと一緒にプロジェクトを企画・立案する**ためのガイドです。

### 使い方

1. このテンプレートをコピーして、実際のプロジェクト名に変更
2. 各セクションを順番に埋めていく
3. AIツール（Claude Code、GitHub Copilot等）と対話しながら詳細化
4. 完成したら `PROJECT.md` と `MASTER.md` に情報を転記

### 関連ドキュメント

プロジェクトの特性に応じて、以下のテンプレートも活用してください：

- [INCEPTION_DECK_TEMPLATE.md](./INCEPTION_DECK_TEMPLATE.md) - アジャイル形式の10の質問でプロジェクトの方向性を明確化
- [USECASE_TEMPLATE.md](./USECASE_TEMPLATE.md) - 機能要件をアクター視点で詳細整理

## 3. 機能要件の洗い出し

### 3.1 コア機能（MVP: Minimum Viable Product）

**Q: 最初のリリースで絶対に必要な機能は？（3-5個）**

優先度の高い順に記入してください：

1. **機能名**: アドベンチャーゲームを実行するロジック
   - **内容**: とあるフォーマットに基づいてシナリオを分岐させて進行するアドベンチャーゲームロジック
   - **理由**: MCPと連携してアドベンチャーゲームを実現するための前提となるロジック

2. **機能名**: ゲームロジックのMCP実装
   - **内容**: ゲームロジックにMCP経由でアクセスできるようにする
   - **理由**: 生成AIがゲームロジックを参照できるようにするため

3. **機能名**: 生成AIと対話してゲームを進行させるAgent Skill
   - **内容**: ゲームをAntigravityやCursorなどのエディタ上で遊べるようにする
   - **理由**: 生成AIとの対話という新しい体験を実現するため

### 3.2 拡張機能（Phase 2以降）

**Q: 余裕があれば追加したい機能は？**

   - **理由**: ゲームのバリエーションを増やすため

### 3.3 対象外（やらないこと）

## 6. リスクと課題

### 6.1 想定されるリスク

**Q: プロジェクトを進める上で心配なことは？**

| リスク                 | 発生確率 | 影響度 | 対策案                               |
| ---------------------- | -------- | ------ | ------------------------------------ |
| ゲームが破綻する・動かない | 高       | 高     | AIの幻覚を防ぐため、状態管理はInkエンジン側に一任する |
| シナリオが面白くない             | 高         | 高       | 最初に小規模なPoC（1部屋からの脱出）を作り面白さを検証する |
| シナリオを作る手間が莫大すぎ              |  中      | 高      | Ink言語を活用し、構造化されたシナリオ作成を効率化。到達不可能性などはコンパイラで検知する |
| ナビゲートするAIがポンコツ              |  中      | 高     | プロンプトで厳格にルールを縛り、現在見えている情報のみをAIに渡す |

### 6.2 技術的な課題

**Q: 技術的に不安な点は？**

- [ ] Ink言語自体の学習コスト（ただし独自の複雑な実装を保守するよりは遥かに安全）
- [ ] MCPサーバー（TypeScript）とInkJS（状態管理）のシームレスな繋ぎ込み
- [ ] 生成AIのプロンプトエンジニアリング（AIを優秀なゲームマスターにするための調整）

## 8. 次のステップ

### 8.1 このテンプレートを埋めたら

1. **AIと対話してブラッシュアップ**

   ```
   Claude Codeなどに以下のように依頼：

   「この企画書をレビューして、以下を教えてください：
   - 曖昧な部分や詰めが甘い部分
   - 追加で検討すべき要件
   - 技術選定のアドバイス
   - リスク対策の提案」
   ```

2. **PROJECT.mdに転記**
   - ビジョン、ステークホルダー、要件定義などを `01-context/PROJECT.md` に整理

3. **CONSTRAINTS.mdに転記**
   - 技術的制約、予算、スケジュールなどを `01-context/CONSTRAINTS.md` に整理

4. **MASTER.mdの作成**
   - 技術スタックとコーディング規約を `MASTER.md` に記載

5. **ARCHITECTURE.mdの設計**
   - システム設計を `02-design/ARCHITECTURE.md` に記載

### 8.2 推奨される進め方

```
企画書（このファイル）
    ↓
PROJECT.md + CONSTRAINTS.md 作成
    ↓
技術選定 → MASTER.md 作成
    ↓
システム設計 → ARCHITECTURE.md 作成
    ↓
実装開始
```
