# PUZZLE_RECIPE_LOCKED_ROOM.md - 密室トリックの作り方レシピ

密室トリック（howdunit）を、adventure-mcp の純メニュー＋状態秘匿で作るための再現
レシピ。難易度モデルやジャンル全体は [PUZZLE_DESIGN.md](./PUZZLE_DESIGN.md) を参照。

- 完成サンプル: `engine/assets/sealed_room.ink`（氷トリック）
- スモークプレイ: `engine/src/domain/services/sealed-room.smoke.spec.ts`

## 0. 密室トリックの論理エンジン

密室は「誰が（whodunit）」ではなく「**どうやって（howdunit）**」を当てる謎。論理構造は
**アブダクション（最良説明推論）＋消去法**である。

> 不可能に見える観測事実の集合に対し、**それを矛盾なく説明できる機構がただ一つ**残る
> ように作る。whodunit の「唯一解の容疑者」を、「全痕跡と整合する唯一の機構」に置き
> 換えたものと考えてよい。

使う論理メソッド（[PUZZLE_DESIGN.md](./PUZZLE_DESIGN.md) §3.3 と対応）:

| メソッド | 中身 |
| --- | --- |
| M1 仮定の棚卸し→1つだけ否定 | 「不可能」を支える前提を列挙し、ちょうど一つを偽にする（残りとは整合） |
| M2 消去法による唯一整合解 | 候補機構を並べ、各手がかりが矛盾するものを落とす。残るのは一つ |
| M3 観測と仮説の矛盾発見 | 痕跡が素朴な解釈と食い違い、再解釈を強制する |
| M4 時間のすり替え | 「密室になった瞬間／犯行時刻」が偽装されている |
| M5 痕跡からの機構逆算 | 物理的痕跡（水たまり・糸くず・傷）から機構を復元する |

## 1. レシピ（トリックから逆算する5ステップ）

密室は**先に機構を決め、その指紋になる手がかりを置く**のが作りやすい。

1. **トリックを1つ選ぶ。**
   例: 氷トリック＝閂を氷で斜めに支えて外へ出て扉を閉める。氷が溶けると閂が自重で
   落ち、内側から施錠されたように見える。
2. **不可能の前提を列挙する（M1）。**
   「閂は内側から下りている＝外から下ろせない」「窓は鉄格子＝人は通れない」
   「他に出口がない」。
3. **そのトリック固有の指紋を1つ置く（M5）。**
   閂の真下の床の小さな水たまり（＝溶けた氷）。**この機構だけがそれを説明する**こと。
4. **対立候補を用意し、各々を別の手がかりで否定できるよう調整する（M2）。**
   - 共犯者が室内に潜伏 → 「埃に乱れがない」で否定
   - 隠し通路（密室は偽り）→ 「窓は格子・壁に通路なし」で否定
   - 糸トリック → 「糸くずも擦り傷もない」で否定
   結果、全手がかりと矛盾しないのは氷トリックだけ＝**機構の唯一性**が成立。
5. **フェアさを確認する。** 指紋（水たまり）と各否定手がかりが、プレイヤーが調査で
   到達できる場所にあること。指紋がなければ「当てずっぽう」になり不公平。

### 良問チェックリスト

- [ ] 全手がかりと矛盾しない機構が**ちょうど一つ**か（唯一性）
- [ ] 正解機構には**裏づけの指紋**があるか（当てずっぽう防止）
- [ ] 各対立候補は**特定の手がかりで明確に否定**できるか
- [ ] どの手がかりも調査で**到達可能**か（隠しすぎて詰まないか）
- [ ] 素朴な解釈（密室は本物）に留まると解けないが、再解釈で解けるか（M1/M3）

## 2. 実装の型

> **最優先**（[PUZZLE_DESIGN.md](./PUZZLE_DESIGN.md) §0）: 解は「手口を N 択で選ぶ」
> より「**世界への働きかけで実演する**」を優先する。例えば氷トリックなら、機構を
> 当てさせて終わりにせず、**水たまりに気づく → 閂の隙間に手を伸ばして鍵を取り返す**
> という"行為"で理解を証明させると、調査の手応えと一貫したアドベンチャー体験になる。
> 下記の N 択「結論づけ」は、行為で表しにくいときの**代替**と位置づける。

### 型A（推奨）: 行為で解を実演する

指紋（水たまり）を見つけたうえで、**機構を逆用する行為**を選べるようにする。

```ink
// 「氷で支えた＝閂の隙間に余地がある」と気づけた者だけが取れる行動。
// 手口を言葉で選ぶのではなく、世界に働きかけて結果を得る。
+ {ev_water && ev_bolt} [閂の隙間に手を伸ばし、奥を探る]
    水で緩んだ隙間の奥に、犯人が隠した鍵が指に触れた。
    ~ has_key = true
    -> hub
```

### 型B（代替）: 論破で機構を絞り込む

行為で表しにくい場合のみ、N 択で機構を結論づける。一発勝負（終端バッドエンド）では
なく、**誤りは手がかりで論破して差し戻す**型にする（学習が進み、デッドエンドにも
ならない）。同梱 `sealed_room.ink` はこの型B を採用している（最小例として読みやすい
ため）。

```ink
=== conclude ===   // 機構を N 択でコミット
+ [共犯者が室内に潜んでいた] -> r_accomplice
+ [隠し通路から出た]         -> r_passage
+ [糸で閂を外から下ろした]   -> r_thread
+ [閂を氷で支え、溶けて落ちた] -> r_ice
+ [まだ分からない（戻る）]    -> hub

=== r_thread ===   // 対立候補は手がかりで論破して差し戻す
{ ev_thread:
    糸を使ったなら糸くずか擦り傷が残るはず。だが何もない。その手口ではない。
- else:
    糸を疑うなら、まず糸の痕跡を探すべきだ。
}
-> hub

=== r_ice ===      // 正解は「裏づけの指紋」を見つけて初めて確定する
{ ev_water:
    閂の真下の水たまりは溶けた氷だ。…全ての辻褄が合うのはこれだけだ。
    ~ has_key = true
- else:
    氷だと？ それを裏づける痕跡をまだ見ていない。当てずっぽうでは納得されない。
}
-> hub
```

ポイント:

- **指紋ゲート**: 正解機構は裏づけ手がかり（`ev_water`）を持つときだけ確定。これで
  「N 択を片端から選ぶ」総当たりを抑える。
- **論破は手がかり条件で**: その否定に必要な手がかりを未取得なら「まず調べよ」と促し、
  取得済みなら明確に否定する。フェアな誘導になる。
- **状態は Ink に**: 手がかりは真偽フラグ、機構の確定で `has_key`（または専用フラグ）。
  AI は痕跡を情景として語るだけで、正解は言わない。

### 難易度を上げるレバー（[PUZZLE_DESIGN.md](./PUZZLE_DESIGN.md) §3.4-3.5 と対応）

- **指紋を間接化**: 「水たまり」を直接見せず、「閂の下だけ埃が固まっている（乾いた泥）」
  のように一段ひねる（連想の飛躍）。
- **多層の仕立て上げ**: 浅い手がかりは糸トリックを示唆する罠にし、精査で覆す。
- **M4 を足す**: 時刻の偽装（犯行は密室化の前）を絡め、howdunit に whodunit/アリバイ
  崩しを重ねる（独立なら加算・相互依存なら一つの大きな謎に）。
- **誤答コスト化**: 学習用なら論破差し戻し、難問なら一発の終端バッドエンドにする。

## 3. トリックの分類（M1 の否定先カタログ）

密室トリックの代表 6 分類を軸に、adventure-mcp 文脈への
翻案・指紋（裏づけ手がかり）・否定する前提を整理する。指紋がそのトリックを一意に
指すように置くのが要点（§1 ステップ3-4）。

| # | トリック | 翻案（盗難＝鍵が密室から消えた） | 指紋（裏づけ手がかり） | 否定する前提 |
| --- | --- | --- | --- | --- |
| T1 | 複雑な鍵の仕掛け（機械トリック） | 扉に仕込んだ装置が後から作動し施錠 | 仕掛けの部品・ばね跡・不自然な細工 | 「内側からしか施錠できない」 |
| T2 | 別場所の鍵（場所・対象の錯誤） | 施錠されていたのは別室／盗まれたのは別物 | 見取り図の矛盾・部屋の取り違え | 「ここが現場だ」 |
| T3 | 自己施錠の偽装（自殺・事故・自作自演） | 被害者/持ち主が自分で隠し、密室に見せた | 持ち主の不審な振る舞い・遺留品 | 「外部の侵入者がいる」 |
| T4 | 痕跡の隠蔽（消去トリック） | 出入りの跡を消した（掃いた・上書き） | 不自然に綺麗・掃いた筋・新しい埃 | 「痕跡がない＝誰も入っていない」 |
| T5 | 犯人の隠匿（室内潜伏） | 発見時、犯人がまだ室内に隠れていた | 隠れ場所・埃の乱れ・温もり | 「室内に誰もいない」 |
| T6 | 証言による虚構（偽証） | 共謀者が「密室だった」と偽証 | 証言同士の食い違い・物証との矛盾 | 「証言は正しい」 |
| T7 | 施錠の遠隔操作（補完） | 糸・磁石・**氷**で外から閂を下ろす | 糸くず／傷／**水たまり**（氷） | 「外から閂は下ろせない」 |
| T8 | 時間の錯誤（補完・M4） | 密室化の前後に犯行をすり替え | 時刻の矛盾（鐘・蝋・砂時計） | 「犯行＝密室成立の瞬間」 |

> T1-T6 は出典記事の 6 分類、T7-T8 は施錠の遠隔操作・時間錯誤の補完。同梱サンプル
> `sealed_room.ink` は **T7（氷）** を採用している。

## 4. トリックの自動選択（指示されたとき）

「密室を作って（トリックはおまかせ）」のように指示された場合、次の手順でトリックを
**自動選択**する。プレイ体験を損なわないため、**選んだトリックや答えはユーザーに告げず**、
シナリオに実装する（ネタバレ防止）。

### 選択の入力（あれば反映、なければ既定）

- **舞台・テーマ**: 例「氷の遺跡」なら T7（氷）が映える、「館もの」なら T1/T6 など。
- **目標難易度**: 易→指紋を明示・候補少なめ／難→指紋を間接化・多層フレーム・T8 を合成。
- **AI 相性**: 対話で解かせたいなら T3/T5/T6/T7（情景と痕跡で語れる）を優先、
  T1（機械）は描写が難しく低め。
- **既出回避**: 直近で使ったトリックは避け、引き出しを回す。

### 手順

1. 上記入力で候補（T1-T8）を絞り込む。指定がなければ全 8 種を候補にする。
2. テーマ適合度と AI 相性で重み付けし、既出を除いて**1つ選ぶ**（迷えば T7→T5→T6 の順で
   無難）。
3. 選んだトリックで §1 のレシピ（指紋を置く・対立候補を否定可能にする・唯一性を確認）を
   実行し、`engine/assets/` に実装する。
4. §5 のスモークプレイで「正解機構で解決」「対立候補は論破」「裏づけ未取得では未確定」を
   検証する。
5. ユーザーには**完成と遊び方だけ**を伝え、トリックの正体は伏せる（求められたら別途開示）。

### 既定の重み付け（指定なしのとき）

AI 相性◎を厚めに: T7・T5・T6・T3 を高め、T2・T4 を中、T1・T8 を低め（描写・実装が
重いか作業的なため）。難易度指定があれば指紋の明示/間接でレバー調整する（§2）。

## 5. 検証（スモークプレイ）

- 正解機構を、裏づけ手がかりを得たうえで選ぶと解決・クリアに到達する
- 対立候補はそれぞれ論破され、差し戻される（デッドエンドにならない）
- 裏づけ未取得では正解機構も確定しない（当てずっぽう防止）
- （論理の厳密さを期すなら）全手がかりに整合する機構が一つだけであることを別途確認

実装例の検証は `engine/src/domain/services/sealed-room.smoke.spec.ts` を参照。
