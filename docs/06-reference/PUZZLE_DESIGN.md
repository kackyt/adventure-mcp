# PUZZLE_DESIGN.md - 謎解きの設計リファレンス

adventure-mcp における「謎解き要素」の設計知見をまとめる。お使い（fetch）ではなく
**プレイヤーが推理して解く謎**を、状態秘匿アーキテクチャ（選択肢メニュー＋サーバ側
バリデーション）を壊さずに作るための、定義・難易度モデル・ジャンル別の引き出し・実装
パターン・検証手順を扱う。Ink 構文や落とし穴は [INK_GUIDE.md](./INK_GUIDE.md) を併読。

関連 Issue: #12（謎の定義・引き出し）、#11（難易度カタログ）、#13（自由入力の拡張）。

## フレームワーク全体像（地図）

シナリオは **4 つの次元＋横断原則** で組み立てる。各次元は独立しており、混同しない
（特に「ボリュームを盛れば難しくなる」という誤解を避ける＝難易度とボリュームは別物）。

```text
            ┌─ ジャンル（コマンド体系・勝利条件）
シナリオ ＝ ┤─ トリック（引き出しの合成）── 難易度（合成の仕方＝レバー）
            └─ ボリューム（場所数 × フラグ数 ＝ 状態量）
   ×（横断）体験原則：探索/試行錯誤を失わない・解は実演・コマンドは固定
```

| 次元 | 正体 | 決めること | 該当節 |
| --- | --- | --- | --- |
| **ジャンル** | コマンドパレット＋勝利条件 | どう操作するか（しらべろ/つかえ/つきつけろ…）・クリアとは何か（脱出/犯人特定/目的達成） | §6 |
| **引き出し** | 謎の部品（型） | どの謎の素材を使うか（連想・証拠解釈・論理・嘘つき・アリバイ・水平思考・換字…） | §4 |
| **トリック** | 引き出しのインスタンス化／合成 | 具体の謎（例: 氷トリック＝「施錠の遠隔操作」、仕立て上げ＝「証拠解釈＋水平思考」）。**難易度はここで決まる** | §2 / §4 / 密室レシピ |
| **難易度** | トリックの作り方＝レバー | 推論の深さ・連想の飛躍・ミスリードの質・相互依存（加算/乗算）・伏線の距離。ジャンルにもボリュームにも依らない | §3 |
| **ボリューム** | 状態量 | 場所数 × フラグ数（＋対象/人物/アイテム数・手数）。難易度とは独立 | #11 |
| **（横断）体験原則** | 体験の土台 | 探索/試行錯誤を失わない・解は宣言でなく実演・コマンドの増減で情報を与えない | §0 |

要約: **ジャンル＝どう操作するか／引き出し＝謎の素材／トリック＝素材の組み立て（難易度が
決まる）／ボリューム＝どれだけの量か**。

## 0. 最優先原則：探索と試行錯誤を失わない

**「答えを選ぶゲーム」にしない。** アドベンチャーの醍醐味は、世界をいじり、試行錯誤
しながら証拠・手がかりを見つけていく**過程**にある。謎がどのジャンルであっても、これを
削ってはならない。

- **解は「宣言」でなく「実演」する。** 「犯人は誰か」「手口は何か」を N 択メニューから
  選ばせる形を最終手段とし、原則は**世界への働きかけで解を示す**。
  - 連想/物理: 油→火を「選ぶ」のでなく**実際に順に行う**（`frozen_key` の型）。
  - 推理（whodunit）: 「カイを選ぶ」でなく**見つけた証拠をその人物に突きつける**
    （古典アドベンチャーの「アイテムを対象に使う」操作）。
  - 密室（howdunit）: 「氷トリックを選ぶ」でなく**閂の隙間に手を伸ばして鍵を取り返す**
    等、理解を**動作で証明**させる。
- **探索そのものを面白くする。** 調べる対象を多く、試した行為に世界が反応し（失敗にも
  フェアな手応え）、手がかりは「ボタンで一覧表示」でなく**いじって発見**させる。
- **コマンドの増減で情報を与えない（ネタバレ防止）。** 「気づくと解決コマンドが新たに
  出現する」appearance-gating は、**出現そのものが答えのヒント**になってしまう（特に
  CLI のように選択肢を一覧表示する UI）。MCP（AI 語り）はメニューを出さないため漏れに
  くいが、本番でも堅牢にするには出し方を次の2方式に寄せる:
  - **結果ゲート（outcome-gating・推奨）**: 解決行動は**常にメニューに置く**（自然な
    行為として）。**成否だけ**を理解状態で出し分ける。気づく前に試しても空振り
    （「指は入るが奥には何もない」）、手がかりが揃って初めて成功する。コマンド集合が
    変化しないので漏れない。
  - **アイテム制／汎用コマンド**: 「使う／触れる／動かす」を**固定の動詞×対象**で常設し、
    解は (動詞, 対象) の組み合わせにする。新コマンドが湧かず、謎は「どれをどこに使うか
    を自分で思いつくこと」になる。
- 要は **解の存在を常に晒しておき、"正しい組み合わせ/タイミングを思いつくこと"を謎に
  する**。コマンドの**ラベルも答えを明かさない**自然な行為にする（「氷トリックで解く」
  でなく「閂の受け穴に手を入れる」）。
- **N 択コミットは"最後の確認"に留める。** どうしても結論の宣言が要る場合でも、それは
  豊かな調査の総仕上げであって、ゲームの中身そのものにしない。

> この原則は §2 の (B)創発／(C)手がかりゲートの精神そのもの。推理・論理系のジャンルは
> 「調査→N択で結論」に退化しやすいので、結論を**在中の行為（証拠の提示・仕掛けの操作）**
> に置き換えられないか常に検討する。

## 0.5 物語接地（diegetic）：仕掛けは世界から生やす

謎の法則・解錠条件・組み合わせの規則は、**世界の理（物理・常識・設定）から必然として
立ち上がる**ように作る。「なぜそうなるのか」を世界の中で説明できない**恣意的なゲーム都合**に
しない。仕組みが良くても、接地していなければプレイヤーは世界を信じられない。

- **反例 ↔ 正例**（同じ融合フレーム・接地だけが違う）:
  - ❌ `apothecary_brew`: 「毒の調合が揮発すると錠が合う」。なぜ毒を調合するのか、なぜ揮発で
    錠が合うのかが世界で説明できず、「めちゃくちゃな小説」になる。
  - ✅ `legendary_forge_cmd`: 「守護者を倒すために得物を鍛える」。**銀の弱点・柄がなければ
    振るえぬ・龍油は刃を鈍らせる**——法則がすべて世界の理として腑に落ちる。
- **接地の緊張**: 帰納に要る「非自明な法則」は恣意的に見えやすい。逆に自明な法則は発見
  （帰納）にならない。狙いは **「非自明だが、解けた瞬間に"世界的に必然だった"と腑に落ちる」**。
  鍵を組み立てる／凶器の組み合わせ（銃と銃弾）／伝説の装備を鍛える、のように
  **「世界で何が機能するか」に法則を一致させる**と、帰納が"理の発見"になる。
- **自己検知できない前提に立つ**: AI は自分の物語の破綻を自力で検知しにくい（知識の呪い）。
  難易度を自己採点できないのと同じく、整合性も自己採点できない。**全自動生成では、前提・
  動機・「なぜその仕掛けが世界で成立するか」を人間が一読して弾く関所**を必ず置く
  （`.rulesync/skills/ink-scenario-creator/SKILL.md` の整合チェック）。

## 1. 「謎」の操作的定義（謎 ≠ お使い）

ある仕掛けが「謎」であるには、次の3条件を**すべて**満たす。

1. **複数要素の統合 or 非自明な順序** — 単一行動で解けず、別々に発見する手がかりの
   組み合わせ、または非自明な手順を要する。
2. **ノンスポイラー** — 正解の選択肢ラベル自体が答えを明かさない（後述の (B)(C)）。
3. **失敗可能かつフェア** — 素朴に進むと解けない経路があるが、提示済みの手がかりから
   論理的に解ける（理不尽な総当たり強制をしない）。

単純な key-lock（「調べる→鍵→開ける」）は条件1・3を満たさず**謎ではない**。
反例サンプル: `docs/06-reference/scenarios/room_escape.ink`。

## 2. ノンスポイラーのコアモデル

選択肢に答えを並べた瞬間に総当たりでネタバレする。これを避ける2モデルを基本とする。

- **(B) 創発パズル** — 個々の選択肢はすべて正当な「行動」。謎の本体は「どの行動を・何を
  持って・どの順で」行うか。ラベルは答えを言わない。
- **(C) 手がかりゲート** — 正解の選択肢は、必要な手がかりが揃うまで `{条件}` で出現
  させない。メニューに答えが漏れない。

```ink
// (B): ラベルは行動のみ。正解は "油→点火" の順序（プレイヤーが推理）
+ {has_oil && not oil_applied} [氷に油をかける]
    ~ oil_applied = true
    -> room
+ {has_torch} [松明をかざす]
    { oil_applied: ~ ice_melted = true  // 油があれば溶ける
    - else: 炎は弱く、氷は溶けない。 }   // フェアな失敗ヒント
    -> room
// (C): 「鍵を取る」は氷が溶けるまでメニューに出ない
+ {ice_melted} [現れた鍵を取る]
    ~ has_key = true
    -> room
```

> **完全一致の自由入力（暗証番号・記号列の入力）はスコープ外**＝エンジン/MCP 拡張
> （#13）。曖昧な自然言語リドル（同義語・表記ゆれ判定）は採用しない。

## 3. 難易度モデル

### 3.1 難易度 ≠ 手数・謎の数

謎・アイテム・部屋を増やして手数を伸ばしても、各々が個別に自明なら「長いだけ」で
難しくならない。**「謎の数・アイテム数・探索範囲・手数」はボリューム（#11）であって
難易度ではない**。

### 3.2 難易度には2つの独立軸がある

| 軸 | 中身 | 純メニュー単独で出せるか |
| --- | --- | --- |
| **推論の深さ**（意図した解の巧妙さ） | 連想の飛躍／証拠統合・矛盾・消去法 | ◯ |
| **総当たり耐性**（しらみ潰しで解けない） | 答え空間が巨大／誤答にコスト | △（小規模N択は総当たり可） |

### 3.3 難しさの2つの“質”

- **手続き的／組合せ**（例: 論理グリッド）— 帳簿付けの量・場合分けの深さ。**転写・背理法
  など確立した解法アルゴリズムがあり、作業的**。AI が代行できてしまう。
- **ひらめき／水平思考**（例: 連想・再解釈）— 連想の距離・前提の裏切り。**汎用解法が
  存在しない**「アハ体験」。小規模だと総当たり可だが、AI にも肩代わりされにくい。

→ **単一の「最強の難しさ」は存在しない**。両者はトレードオフ（手続き的は AI が解け、
ひらめきは規模で総当たりに弱い）。adventure-mcp は「AI と対話しながら解く」体験が核な
ので、**ひらめき型・証拠解釈型を主軸**にし、論理グリッドは作業性に注意して使う。

### 3.4 連想型の難易度レバー（推論の深さ）

| レバー | 易しい | 難しい |
| --- | --- | --- |
| 推論ステップ数（手がかり→解の距離） | ほぼ直接（1段） | 中間命題を自分で立てる多段 |
| 連想の飛躍 | 常識的（油は燃える） | 遠い／別用途の転用 |
| ミスリードの質 | なし／見え透いた | 一番素直な行動が罠 |
| 同時制約 | 単一 | 順序・排他・リソースを同時管理 |
| 時間的・空間的距離（伏線） | 直前・同室で完結 | 序盤の"用済み"が遠い場面で再浮上 |
| 手がかりの明示度（精査テキスト） | 結論まで言う | 事実だけ示し結論は推理させる |

### 3.5 純メニューで総当たりを抑える技

| 技 | 効果 |
| --- | --- |
| 段階ゲート調査 | 前の発見が次の調査を解禁。決め手は最後にしか出ない |
| 仕立て上げの罠 | 一番それっぽい証拠が無実の者を指す。素朴な告発は失敗 |
| 決め手の秘匿 | 真の決め手を発見しないと正解の組み合わせを作れない |
| 誤答コスト | 無実を名指す等の誤答は終端バッドエンド（ソフトロックにしない） |
| 多次元の同時確定 | 犯人×手口×決め手を一括コミット。答え空間が積で膨らむ |
| 結論空間の構造化 | 単一N択でなく「配置/割当」を提出（4!=24 等） |

### 3.6 難易度の合成則（加算 vs 乗算）

複数の小推理を組み合わせたときの効き方は、**部品が独立か相互依存かで決まる**。

- **独立**（手がかりを共有しない）かつ**一括コミット・部分正解の手応えなし**:
  - 総当たり耐性 ≈ **∏ nᵢ（乗算）** ／ 賢い解き手の負荷 ≈ **Σ Rᵢ（加算）**。
  - この非対称が「考えれば解けるが当て勘では無理」の幅を生む（最大効率）。
  - **ただし独立＝「一つの謎」ではなく、複数の謎を連続で解いているだけ**になる。
  - 検証 PoC: `docs/06-reference/scenarios/theft_combined.ink`。
- **相互依存**（手がかりが次元をまたいで絡む）→ それは**一つの謎**。
  - 一方向の連鎖なら線形（解けるが一本道）。循環して入口がないと背理法が要り、
    推論は超加算（組合せ的）＝本物の難しさ。
  - 検証 PoC: `docs/06-reference/scenarios/theft_logic_interlock.ink`（4要素・前方連鎖。
    背理法必須の"真の難"は要素数 5〜6 や不足決定が要る）。

> 注意: 乗算が効くのは「独立かつ各次元が一発勝負で必須」のとき。途中で部分正解が
> 分かると次元を固定して残りを総当たりでき、乗算は加算に崩れる。

### 3.7 難易度づくりの失敗モード（実証済み）

「難しくしたつもり」が易しくなる典型。設計時にこの5つを潰す。

| 失敗モード | 何が起きるか | 対策 |
| --- | --- | --- |
| **機械的消去で解ける** | 小規模な論理グリッド／割当は、消去法という確立アルゴリズムで作業的に解け、深さがない | 推論の深さ（§3.4）か総当たり耐性（§3.5）を足す。要素4程度の単純グリッドは「難しい」と呼ばない |
| **鍵の"提示"＝再認で解ける** | 正解の着想を選択肢ラベルや単独の手がかりに直接書くと、プレイヤーは「思いつく（生成）」でなく「選ぶ／読む（再認）」だけで済み、易化する | 鍵を単独の手がかりに書かず**複数に分散して統合を強制**。着想そのものを選択肢に並べない（§0・§2(B)(C)） |
| **反復で飽きる（またあれか）** | 型を知ると解法が手に入り難易度が蒸発する。同じジャンルを繰り返すと「またあれか」になる | 単一型に頼らず**多様性＋合成**。または毎回その場で生成が要る型（深い演繹・帰納・融合）を使う |
| **最終が推測可能** | 正解が少数かつ自然な見た目だと、法則を見抜かずとも当てずっぽで通る | 解空間を広げる／一度きりの誤答コスト／**"新しい複数ケースへの適用（審判）"で生成を強制**（帰納型の肝） |
| **法則が恣意的** | 仕掛けが世界と無関係（揮発で錠が合う等）だと、難しさ以前に「めちゃくちゃ」になる | §0.5 物語接地：法則を世界の理から生やす |

> 共通の核: **純メニューでプレイヤーが"再認/選択"で済むものは易しい。"生成"（自分で着想・法則・統合を作る）を強いる設計だけが難しい。**

## 4. 謎のジャンル別カタログ（引き出し）

各ジャンルの最小 PoC（`.ink` ソース）は `docs/06-reference/scenarios/` に参照保管して
ある（参照資料のためコンパイル対象外）。AI 相性は「AI と対話しながら解く」体験への向き。

| --- | --- | --- | --- | --- |
| お使い（反例） | room_escape | — | 調べる→鍵→開ける | — |
| 連想/物理ギミック | frozen_key | 連想の飛躍・順序 | (B)(C)。アイテム＋順序フラグ | ◎ |
| 推理/証拠解釈 | stolen_key_mystery | 段階調査・仕立て上げの罠 | 段階ゲート＋決め手の秘匿＋犯人×証拠 | ◎ |
| 制約 intersection | theft_intersection | 属性の積集合・多層フレーム | 各手がかりが容疑者を1人ずつ消去 | ○ |
| 論理グリッド | theft_logicgrid | 割当の組合せ（4!=24） | 各人の位置を割当→一括提出 | △(作業的) |
| 相互依存ロジック | theft_logic_interlock | クロス制約 | 位置×属性が絡む単一推理 | △ |

| 密室トリック（howdunit） | sealed_room | 不可能の唯一整合する機構 | アブダクション＋消去法。指紋で機構を確定 | ◎ |
| 帰納（隠れた法則の発見） | oracle_runes | 試行→観察→法則の構築（探偵でなく科学者の推論） | 神託に問い、**新しい複数ケースの分類「審判」で適用**。問題数 > 実験回数で丸暗記でなく一般化を強制 | ◎ |
| 帰納×演繹の融合 | legendary_forge_cmd | 演繹で候補を絞り、帰納（実験）が最後の二択を断つ | 手記で2候補へ→鱗片に試して（銀だけ噛む）確定。固定コマンド（つかう→道具→対象）で実演。接地版＝`legendary_forge_cmd`、独自UI版＝`legendary_forge`、非接地の反例＝`apothecary_brew` | ◎ |

> **帰納型の肝**: 法則は「実演で観察できるが、どこにも明記されない」こと。ゴールを「1つの正解配置を作る」にすると推測で通り帰納が飾りになる（`oracle_runes` v1 の失敗）。**学んだ法則を"新しい複数ケース"へ適用させる**（分類審判・別案件への適用）と、推測が封じられ生成が強制される。融合型は「演繹のみ=N候補／法則込み=一意」をスクリプトで機械検証する。

密室トリックの組み立て方は専用レシピ
[PUZZLE_RECIPE_LOCKED_ROOM.md](./PUZZLE_RECIPE_LOCKED_ROOM.md) に詳述。

## 5. ジャンル横断の実装パターン

- **状態は Ink に一任**: 真偽フラグ／`LIST`＋`carries()`／整数コードで管理。AI に状態を
  持たせない。自己ループ部屋の一度きり判定は訪問回数でなく**明示フラグ**で（INK_GUIDE
  の落とし穴参照）。
- **告発・提出は専用ノット**: 犯人 N 択や配置の割当は独立ノットに分け、`{全項目セット}`
  でのみ「提出」を出す。誤答は `-> game_over`/`-> END` の**終端ED**へ（デッドエンドに
  しない）。
- **再挑戦の作法**: 手順系（換字暗号など）の誤りはリセットして再挑戦可に。推理系の
  誤指摘は終端バッドエンドで一発のコストにする、と使い分ける。
- **ヒントは Ink に段階配置**: AI は状態を知らない。手がかりは Ink 内に段階的に埋め、
  AI はそれを口調づけして提示する。

## 6. 汎用コマンドのジャンル別パレット

§0 の「コマンドの増減で情報を与えない」を満たす最も実用的な形が、**ジャンルごとに
固定の動詞セット（コマンドパレット）を用意し、対象だけ差し替える**こと。動詞は終始
変わらず、謎は「どの動詞を・どの対象に」当てるかになる。古典的な探偵アドベンチャーの
固定コマンド体系が範。各パレットは雛形 PoC を持つ。

> **動詞は終止形（「しらべる」）で書く。** 助手に指示する形式のゲームは命令形（「しらべろ」）
> が自然だが、**主人公ひとり**の視点では命令形は不自然。プレイヤー自身の行為として
> 終止形に統一する。

| ジャンル | 固定コマンド（動詞） | 解決の核 | 雛形 PoC |
| --- | --- | --- | --- |
| **探偵・推理** | あたりをみる／しらべる→対象／きく→人物／なにかきく→人物／なにかとる／ばしょいどう→場所／でんわをかける／たのむ／**つきつける→人物→証拠** | 証拠を相手に突きつける | `detective_command.ink` |
| **脱出** | しらべる→対象／とる／**つかう→道具→対象**／おす・うごかす→対象／もちもの／ばしょいどう→場所 | 道具を対象に使う（組み合わせ） | `escape_room.ink` / `escape_mansion.ink` |
| **冒険**（点アンドクリック型） | しらべる→対象／あける・しめる→対象／とる・すてる→対象／**つかう→道具→対象**／たたく→対象／はなす→対象／うごく→場所／もちもの | 探索＋道具の組み合わせ＋移動（明かり等の状態管理・予兆つき罠死） | `adventure_command.ink` |

実装の要点:

- **動詞はトップに常設**。`しらべろ`→対象、`つきつけろ`→人物→証拠 のように、動詞ノットが
  対象のサブメニューへ分岐する（解決専用コマンドを湧かせない）。
- **対象は場所（`loc`）で出し分ける**。`+ {loc == 1} [台座] -> ...` のように現在地で
  サブメニューの選択肢を絞る。場所＝ノット、移動＝divert（各場所ハブに「○○へ移動する」）。
- **誤った動詞×対象はフェアに空振り**（「効果がない」「当てはまらない」）。決め手は
  入手していなければ突きつけられない（§0 の outcome-gating）。
- **インベントリ／戻り先**は `back` 変数で共有ノット（`もちもの`）から元の場所へ戻す
  （`sages_tower.ink` の `observe` 同様）。

## 7. サウンドノベル（分岐物語）ハーネス

探索コマンド型とは別系統。操作は「**読み進める＋分岐選択**」で、謎は必須ではなく
**物語の選択・フラグ分岐・複数エンディング**が主役になる（フレームワークの「ジャンル」
次元の一つ）。Ink は分岐物語のために作られており、weave・gather・変数・複数 `END` で
素直に書ける。雛形 PoC は2種。

### 標準型（`sound_novel.ink`）— 古典的なサウンドノベルの慣習

- **多重エンディング・フローチャート**: 序章 → 分岐選択 → ルート → 複数の `-> END`
  （GOOD / BAD / TRUE）。
- **フラグ蓄積分岐**: 選択で `courage`/`trust` 等が変化し、結末が分かれる。
- **既読エンディング管理＋真ルート解禁**: 結末で `seen_good`/`seen_bad` を立て、両方
  見ると **TRUE ルート**が出現する（「全 ED 回収で特殊ルート解禁」の定番）。
- **タイトルへ戻るループ**: 結末 → タイトルへ戻る。グローバル変数（`seen_*`）は保持
  されるので、周回で解禁が進む。セーブ機構なしでも1セッション内で成立する。

```ink
=== title ===
…
+ {seen_good && seen_bad && not seen_true} [隠された真実をたどる] -> truth_route  // 解禁
+ {seen_good || seen_bad} [もう一度、あの夜へ] -> prologue
=== ending_good ===
【GOOD END】… ~ seen_good = true
-> ending_return    // ending_return -> title へ戻る（フラグ保持）
```

### ザッピング型（`zapping_novel.ink`）— 複数主人公・時刻ハンドオフ（複数視点系）

複数視点型。**視点を常時トグルするのではなく**、物語は一本の流れとして
主人公間を渡り歩く。要所で「ZAPPING」して**別主人公の時刻へハンドオフ**する。

- **時刻ハンドオフ**: 各主人公のシーンは時刻付き（10:00, 10:30…）。シーンの終わりで
  別主人公の時刻へ ZAPPING する（自由な視点メニューではない）。
- **連動 BADEND（この型の肝）**: ある主人公の選択が、**別主人公の物語**を BADEND に
  する。例: サトルが力ずくで止める → 巻き添えで**トメ編 BADEND**／ミオが近道を抜ける
  → 追う羽目になった**サトル編 BADEND**。
- **BADEND は終端＋やり直し**: ソフトロックにしない（「初めからやり直す／受け入れて
  終える」）。全シーンを誰も BADEND にせず通すとクリア。

```ink
=== sato_1000 ===
【刑事サトル ／ 10:00 …】
+ [力ずくで取り押さえる]   … -> bad_tome      // 自分の選択が "別主人公" を BADEND に
+ [穏便に諭す]             … -> zap_to_tome   // 進行 → トメ編 10:30 へ
=== zap_to_tome ===
〜 ZAPPING ／ 露天商トメ ・ 10:30 へ 〜
-> tome_1030
=== bad_tome ===
【BADEND・トメ編】… -> retry   // retry: [やり直す] / [受け入れて終える]
```

> 検証は標準型の「**全分岐・全エンディング網羅**」「真ルート解禁」に加え、ザッピング型は
> 「**連動 BADEND（ある主人公の選択が別主人公を BADEND にする）**」「時刻ハンドオフ」
> 「BADEND からのやり直し」をスモークプレイで確認する
> （参照シナリオ: `docs/06-reference/scenarios/sound_novel.ink` / `zapping_novel.ink`）。

## 8. 検証（スモークプレイ必須）

`pnpm run build:ink` のコンパイル成功だけでは実行時の破綻を検出できない。**生成 JSON を
inkjs で再生**し、最低限を確認する（先頭 BOM を `replace(/^﻿/, "")` で除去）。

- 手がかりから正解ルートに到達でき、クリア（`-> DONE`/`-> END`）できる
- 素朴に進む「失敗ルート」が存在し、そこから挽回できる（デッドエンドにならない）
- バッドエンドがある場合、意図どおり発生し、それも終端ED（ソフトロックでない）
- 論理パズルは**唯一解であること**を別途検証する（総当たりスクリプト等）
- 一度きりイベントが再訪で再発しない

参照シナリオ（`.ink` ソース）は `docs/06-reference/scenarios/` に保管（参照資料・コンパイル
対象外）。**新しいライブシナリオを `engine/assets/` に作るときは、上記をスモークプレイで
検証するテストを必ず添える。**

## 9. 豊かなボリューム・持続的難易度・物語進行の足場（推理もの中心）

PoC 実証で判明した「おまかせ生成が薄くなる」失敗（一巡で全手がかり取得＝10分、消去法で
犯人特定、人物が固定台詞で物語が進展しない）への、構造的な処方。**ボリュームはノット数で
なく「依存の深さ × 物語の節」で作る。**

### 9.1 ボリューム＝段階フェーズ × 多段クリア × 節

| 仕掛け | 内容 | 大ボリュームの目安 |
| --- | --- | --- |
| **段階フェーズ（gated phases）** | 調査を波状に解禁。初動で表層→発見が次フェーズ（新エリア/人物/話題）を解禁→…。**全手がかりを一巡で取らせない** | フェーズ ≥3 |
| **容疑者の多段クリア** | 各容疑者のアリバイ/嫌疑を1手で白黒つけない。証言→裏取り→矛盾→再尋問、と最低2手 | 各容疑者 2手以上 |
| **サブ謎** | 進行をゲートする小謎（ロックされた端末・暗号ログ・消えたICカード等） | サブ謎 ≥2 |
| **進化する証言** | 証拠を突きつける/新事実を得ると、人物の台詞・態度・開示が変わる。再尋問で新情報 | 主要人物に状態依存の台詞 |
| **中盤の再フレーム** | 一度「絵」が変わる（疑っていた構図が覆る）どんでん返し | ≥1 |

> 想定時間 ≒ Σ(各フェーズの調査手数)。**ノットを増やすだけ（フラットな一発入手の山）では
> 尺は伸びない**。短ければフェーズ・サブ謎・進化証言・再フレームを足して反復（拡張ループ）。

### 9.2 消去法に化けさせない（持続的難易度）

- **犯人を「最後の1人」にしない。** タダの除外（カメラで即無罪等）を作らず、各容疑者の排除に
  能動的な裏取りを要求する。
- **仕立て上げを本気で。** 偽の真犯人を、最後の飛躍まで「確信して告発できる」強さにする
  （素朴な消去法はむしろ偽犯人を指す＝真犯人は正の推理でのみ届く）。
- 例: 在館＋動機＋空白時間で早瀬が最有力に見える → そのまま告発すると冤罪ED。真犯人桐生は
  「時計ズレの帰納」という正の推理でしか届かない。**消去では誤り、推理でのみ正解**に設計する。

### 9.3 物語進行（満足感の源）

- **状態依存の台詞**: 証拠提示・新事実で人物の反応が変わる。再尋問で開示が進む。
- **サブプロット・人間関係**: 容疑者同士の関係や過去が、調査の進展とともに少しずつ見えてくる。
- **節（beat）**: フェーズ境界に小さな山場（新事実の発覚・関係の判明・どんでん返し）を置く。

### 9.4 自動生成の限界と共作誘導

おまかせは**コア謎**は出せるが、上記の厚み（特に物語の進展・満足感）と大ボリュームを自力で
十分に出すのは難しい（知識の呪い＝薄さを自己検知しにくい）。**大ボリューム × リッチ物語は
共作モード**（人間が物語・構成を駆動、AI が機構）を推奨。おまかせで作る場合も、完成時に
「尺・厚み・非消去・物語進展」を人間が確認する**関所**を必ず通す。
